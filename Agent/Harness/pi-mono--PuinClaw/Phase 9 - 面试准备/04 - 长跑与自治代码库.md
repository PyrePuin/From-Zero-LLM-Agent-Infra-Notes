# 04 - 长跑与自治代码库

> [!note]
> 这是 PuinClaw 已经做了一半的方向。
> **长跑** = `/goal` 模式让 agent 持续推进
> **自治代码库** = 更激进 — agent 不只是"完成一个 goal",而是**长期管理一个代码库**
> 涉及两篇较难文章:**Extending long-running autonomous coding** + **Towards autonomous codebase operation**。

## 1. 长跑(你已经熟)

```
PuinClaw 的 /goal 已经实现了长跑:
   ┌────────────────────────────────────────────┐
   │ while (true) {                             │
   │   ① 检查 mark_complete / pause / cap       │
   │   ② state.turn += 1                        │
   │   ③ agent.prompt(userKick)                 │
   │   ④ 持久化 state.json + messages.json     │
   │ }                                          │
   └────────────────────────────────────────────┘
```

### 长跑的 5 个关键能力

```
① 持续推进  — 不停跑(prompt loop / followUp / Heartbeat)
② 进度持久  — state.json 跨重启 resume
③ 安全 cap — max_turns + max_cost
④ 优雅中止  — SIGINT → paused 不是 killed
⑤ 上下文治理 — 压缩老消息,防止 context 爆炸
```

## 2. 自治代码库(更进一步)

```
长跑:agent 完成"一个目标" → 结束
自治代码库:agent 长期驻留,持续管理整个代码库

┌──────────────────────────────────────────────────────┐
│ 自治代码库(Autonomous Codebase Operation)          │
│                                                      │
│ Agent 7×24 在代码库里"生活",自动:                  │
│   • 处理 GitHub issue / PR                           │
│   • 修复 bug                                         │
│   • 写测试                                           │
│   • 重构                                             │
│   • 升级依赖                                         │
│   • 回应 CI 失败                                     │
│   • 写文档                                           │
│                                                      │
│ 人在干嘛?                                           │
│   • Review agent 的 PR                              │
│   • 设定优先级                                       │
│   • 处理 agent 升级不到的"战略问题"                  │
└──────────────────────────────────────────────────────┘
```

### 跟长跑的核心区别

| 维度 | 长跑 | 自治代码库 |
|---|---|---|
| 任务来源 | 单个 goal | 持续事件流(issue/PR/CI) |
| 终止条件 | mark_complete | **永不终止** |
| 状态 | 单一 state.json | 多个 task + 优先级队列 |
| 触发 | 用户 /goal | 事件驱动(webhook/cron) |
| 失败 | failed → 退出 | failed → 重试 / 下一个 |
| Context | 一份 transcript | 多个独立 transcript(每 task 一份) |

## 3. 关键文章要点

### 3.1 Extending long-running autonomous coding

```
核心问题:agent 跑久了会"漂移"
   ┌────────────────────────────────────────┐
   │ 漂移类型:                             │
   │  ① 目标漂移 — 忘了最初目标            │
   │  ② 上下文爆炸 — 老 turn 堆太多        │
   │  ③ 错误循环 — 反复尝试同一个错误      │
   │  ④ 死锁 — agent 卡住无进展            │
   │  ⑤ 成本失控 — 跑太久 cost 涨太多      │
   └────────────────────────────────────────┘

应对策略:
   ① 周期性重注入目标(Heartbeat)
   ② 上下文压缩(compaction)
   ③ 死锁检测(N turn 无进展 → 重启)
   ④ Checkpoint + rollback
   ⑤ 双重 cap(turns + cost)
```

### 3.2 Towards autonomous codebase operation

```
从"单 goal 长 agent"到"代码库 agent 团队":

   ┌─────────────────────────────────────────┐
   │ 单 goal 长跑模型                        │
   │   user → /goal → agent 跑完 → 结束     │
   ├─────────────────────────────────────────┤
   │ 自治代码库模型                          │
   │                                         │
   │   GitHub ──→ ┌──────────────┐           │
   │   Issues ──→ │              │ ──→ PR    │
   │   PR     ──→ │ Codebase     │ ──→ PR    │
   │   CI fail ─→ │ Agent Team   │ ──→ PR    │
   │   Cron ────→ │ (always on)  │ ──→ PR    │
   │              └──────────────┘           │
   │                     ↑                  │
   │                  Human review          │
   └─────────────────────────────────────────┘

关键特征:
   • 事件驱动(webhook)
   • 多 agent(team)
   • 永不停歇(always-on)
   • 人在环(human-in-the-loop review)
```

## 5. 关键挑战

```
挑战 1: 上下文管理
   ─→ 老 transcript 压缩(compaction)
   ─→ 关键事实外置(state.json / todos)
   ─→ 多 agent 各自独立 transcript

挑战 2: 失败恢复
   ─→ checkpoint + rollback
   ─→ 死锁检测(N turn 无 progress)
   ─→ 错误分类(可重试 vs 不可重试)

挑战 3: 成本控制
   ─→ 单 task cost cap
   ─→ 月度总预算
   ─→ 优先级(高优 task 多花)

挑战 4: 质量保证
   ─→ Reviewer agent
   ─→ CI 强制
   ─→ Human review 关键 PR

挑战 5: 安全
   ─→ 不能 merge 未 review 的 PR
   ─→ 不能 force push
   ─→ 不能改 CI 配置
```

## 6. 面试要点

### 必答清单

```
Q1: 长 agent 跑久了会什么问题?
A: 5 类漂移 — 目标漂移 / 上下文爆炸 / 错误循环 / 死锁 / 成本失控。
   对应:Heartbeat / compaction / 死锁检测 / checkpoint / cap。

Q2: 长跑 vs 自治代码库区别?
A: 长跑是"完成单个目标" → 终止。
   自治代码库是"长期驻留,事件驱动,永不终止"。

Q3: 自治代码库需要什么基础设施?
A: 事件源(webhook/cron) + 队列 + dispatcher + Agent team + PR 流程 + CI。

Q4: 怎么避免 agent 损坏代码库?
A: ① Worktree 隔离
   ② CI 强制
   ③ Reviewer(自动 + 人工)
   ④ 权限控制(不能改 CI/force push)
   ⑤ 关键 PR 必须 human review

Q5: 上下文爆炸怎么办?
A: ① Compaction(老消息摘要)
   ② 多 agent 各自独立 transcript
   ③ 关键事实外置(state.json)
   ④ 长任务分阶段,每阶段独立 transcript
```
