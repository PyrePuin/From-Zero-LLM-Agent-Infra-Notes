## 1. 概念辨析(必考,L1-L2)

### Q1.1 LLM、Agent、Workflow 三者的区别?

**答题要点**:
- **LLM**:单次输入→输出,无状态、无行动能力。
- **Workflow**:固定 DAG,每一步用 LLM 但路径写死,确定性高。
- **Agent**:LLM **自己决定下一步做什么**(tool / think / finish),路径不固定。

**一句话**:`Agent = LLM + Harness(循环 + 工具 + 状态 + ...)`。

**加分点**:引用 Mitchell Hashimoto 2026-02-05 提法 —— "Agent = Model + Harness",或 OpenAI 官方文章《Harness engineering》的层叠视角。

**陷阱**:别说"Agent 就是智能体"或"会思考的 LLM"——空话。

---

### Q1.2 为什么需要 Agent,单纯的大模型不够吗?

**答题要点**(参考 28 届真题 #2):
- LLM 三大局限:**无行动能力 / 无法与外界交互 / 无法完成多步任务**。
- Agent 用"感知→规划→行动→学习"闭环补齐这三块。
- 工具扩展能力边界(Function Call / MCP / Skills)、记忆扩展时间边界(短期/长期)、循环扩展推理边界(多步)。

**加分点**:能举一个具体场景——"做一道 LeetCode 题,LLM 只能输出代码,Agent 能写测试、跑测试、看报错、改代码"。

---

### Q1.3 Agent 的核心组件有哪些?

**答题要点**:

| 组件 | 作用 |
| --- | --- |
| **模型(LLM)** | 大脑——推理、决策 |
| **提示词(Prompt)** | 角色定义 + 规则 |
| **工具(Tools)** | 手脚——查数据库、执行代码、调 API |
| **记忆(Memory)** | 跨轮/跨会话状态 |
| **Agent Loop** | 主循环——think → act → observe |
| **状态管理(State)** | 任务/子任务/中间结果 |

**陷阱**:别只列"规划/记忆/工具/行动"四件套就完——面试官要听你讲**它们怎么串起来**(数据流)。

---

### Q1.4 Function Calling、MCP、Skills、Agent 是什么关系?

**答题要点**(牛客·小红书一面真题 #4):

| 概念 | 层次 | 解决的问题 |
| --- | --- | --- |
| **MCP** | 协议层 | "接得规范"——标准化工具/资源接入 |
| **Function Calling** | 模型能力 | "调得出来"——输出结构化调用意图 |
| **Skills** | 复用层 | "能力可复用"——封装好的能力单元(如"查工单") |
| **Agent** | 调度层 | "组合成任务闭环"——规划+调用+反馈+迭代 |

**一句话**:MCP 管**通信规范**,Function Calling 管**意图表达**,Skills 管**能力复用**,Agent 管**任务调度**。

**加分点**:提到 MCP 是 Anthropic 2024-11 推出的开放协议,客户端/服务端架构;Function Calling 是 OpenAI 2023-06 推出的模型能力。

---

### Q1.5 Agent 和传统 RPA 有什么区别?

**答题要点**:

| 维度 | RPA | Agent |
| --- | --- | --- |
| **决策方式** | 规则驱动 | LLM 推理 |
| **适应能力** | 流程变就崩 | 容忍模糊输入 |
| **学习机制** | 无 | 反思 + 记忆积累 |
| **错误处理** | 写死 fallback | 自己重试/换路径 |

**陷阱**:别说"RPA 落后,Agent 先进"——要说**各自适用场景**:RPA 适合流程稳定的银行/财务,Agent 适合需求模糊的客服/编码助手。

## 2. Agent 核心原理(L2-L3)

### Q2.1 Plan→Act→Observe→Reflect 在生产中怎么落地?

**答题要点**(牛客·花海真题):

| 阶段 | 输入 | 输出 | 异常处理 |
| --- | --- | --- | --- |
| **Plan** | 用户目标 | 步骤列表 + 工具依赖 | 目标模糊→反问用户 |
| **Act** | 单步 + 工具 | 工具返回结果 | 超时→重试/熔断 |
| **Observe** | 工具结果 | 是否达到终止条件 | 结果格式异常→清洗 |
| **Reflect** | 完整轨迹 | 是否完成 + 是否漏调 | 反思超预算→人工接管 |

**加分点**:
- 强调 Reflect **不能无限循环**,要给预算(最多重试 N 轮)。
- 提到 pi-mono 的实现:`runner.ts` 里 while(true) 主循环 + turn cap(默认 100)+ cost cap(默认 $5)。

---

### Q2.2 ReAct 框架怎么避免"思考与行动脱节"?

**答题要点**(花海真题):
- **脱节根因**:Reason 步骤模型自由发挥,与 Act 实际能力不匹配。
- **三种解法**:
  1. **工具描述结构化**:统一 schema(name/description/parameters/when_to_use)。
  2. **思维链约束**:Reason 必须先说"我现在要调 X 工具,因为 Y"。
  3. **失败负反馈**:Act 失败立刻进 Observe,而不是继续自由推理。

---

### Q2.3 Agent 工具调用 Schema 设计核心是什么?

**答题要点**:
- **name**:动词 + 名词(`search_order` 而不是 `search`)。
- **description**:必须包含 **when_to_use**(什么时候用)+ **when_not_to_use**。
- **parameters**:JSON Schema,字段都有 description + 默认值 + 枚举。
- **return_schema**:输出格式严格定义,方便 Observe。

**陷阱**:Schema 写得越复杂越好——错。**LLM 偏好简单清晰的描述**,字段一多就开始乱填。

---

### Q2.4 Agent 系统中工具调用失败怎么办?

**答题要点**(28 届真题压轴):
- **分级处理**:

```
瞬时失败(网络抖动)
   → 指数退避重试(3 次)

参数错误(LLM 乱填)
   → 提示 LLM 重新选工具/重传参数

权限失败(403)
   → 直接失败,不重试(避免暴力破解告警)

工具不可用(404)
   → 降级到 fallback 工具或返回"能力不可用"

连续失败 N 次
   → 熔断 + 人工接管
```

**加分点**:提到 pi-mono 的 `tryExecute` 包了 `RetryWithBackoff`,codex 的 sandbox 在权限失败时直接 kill 进程。

---

### Q2.5 多步工具依赖(查用户→查订单→查物流)怎么避免重复调用和死循环?

**答题要点**:
- **状态机思维**:把每步结果存到 `state` 里,下一步从 `state` 取参,不重新调。
- **依赖图**:Plan 阶段就画好 DAG,Act 时按拓扑序执行。
- **去重 hash**:对 (tool, params) 计算 hash,本轮已调过直接返回缓存。
- **死循环检测**:记录 (tool, params) 出现次数,超过 2 次强制终止。

---

### Q2.6 Agent 的记忆怎么分层?为什么不能只靠聊天历史?

**答题要点**(小红书真题 #6):

**问题**:聊天历史越长,无关内容越多,影响当前判断 + Token 爆炸。

**三层模型**:

| 层 | 内容 | 时效 | 存储 |
| --- | --- | --- | --- |
| **短期工作记忆** | 当前任务的中间状态 | 任务结束即清 | 进程内 |
| **长期用户记忆** | 偏好/习惯/背景 | 跨会话稳定 | 数据库 |
| **外部事实记忆** | 知识库/文档 | 可检索可追溯 | 向量库 |

**陷阱**:工作记忆和事实记忆**必须分开**,否则一次错误推断会沉淀为长期污染。

**加分点**:提到 claw0 的 `SOUL.md`(长期用户画像)+ pi-mono 的 `pi-memory` 包。

---

### Q2.7 ReAct、Plan-and-Execute、Reflection、Multi-Agent 这几种模式怎么选?

**答题要点**:

| 模式 | 适用 | 不适用 | 复杂度 |
| --- | --- | --- | --- |
| **ReAct** | 简单工具调用 | 多步规划 | ⭐ |
| **Plan-Execute** | 步骤明确 | 探索性任务 | ⭐⭐ |
| **Reflection** | 需要质量保证 | 时延敏感 | ⭐⭐ |
| **Multi-Agent** | 角色天然分工 | 短平快任务 | ⭐⭐⭐⭐ |

**陷阱**:别一上来就说"我用 Multi-Agent"——Multi-Agent 通信成本高,绝大多数场景单 Agent + Reflection 就够了。

## 3. Harness 工程(L3-L4,面试亮点)

### Q3.1 什么是 Harness Engineering?

**答题要点**:
- **起源**:Mitchell Hashimoto(HashiCorp 创始人)2026-02-05 文章《The Agent Era》提出 `Agent = Model + Harness`。
- **OpenAI 官方背书**:2026-02-11 文章《Harness engineering: leveraging Codex in an agent-first world》。
- **核心**:Harness **不是组件**,是夹在 LLM 模型层和应用层之间的**所有工程**。

**一句话**:**Harness = Agent Loop + Tools + Context Eng + Sandbox + Memory + Multi-Agent + Session + Evaluation**。

---

### Q3.2 Harness 的 6 层是什么?

**答题要点**:

| 层 | 作用 | 关键技术 |
| --- | --- | --- |
| **上下文精细化** | 喂给模型最相关的信息 | RAG / 压缩 / 优先级 |
| **工具系统** | 给模型手脚 | Function Call / MCP / Skills |
| **执行编排** | 决定下一步做什么 | Agent Loop / Plan-Execute |
| **记忆与状态** | 跨轮/跨会话连续 | 短期/长期/向量记忆 |
| **评估观测** | 知道 agent 跑得怎么样 | Trace / Eval / DSPy |
| **约束恢复** | 安全兜底 | Sandbox / Rate Limit / 熔断 |

**加分点**:每层都能举一个 PuinClaw 里的对应实现(例如 上下文精细化→pi-ai 的 convertToLlm)。

---

### Q3.3 Harness 时代工程师的角色是什么?

**答题要点**(OpenAI 文章核心):
- **从"写业务代码"→"设计 agent 协作环境"**。
- 工程师不再写 CRUD,而是**搭一个让 agent 高效产出的工作台**。
- 类比:从"流水线工人"→"流水线设计师"。

**金句**:OpenAI 原文 — "Redefining the role of the engineer"。

---

### Q3.4 Harness 工程的 5 大真实挑战是什么?

**答题要点**:

| 挑战 | 现象 | 应对 |
| --- | --- | --- |
| **Context Rot** | 上下文越长,模型表现越差 | Context Reset / 主动压缩 |
| **Self-eval bias** | 模型评估自己偏向"已完成" | 外部 verifier / 人工抽检 |
| **Repeated failure** | 失败后陷入相同循环 | 检测重复模式 + 强制换路径 |
| **AGENTS.md bloat** | 项目说明越积越多 | Progressive Disclosure(<100 行) |
| **AI slop** | agent 生成大量似是而非代码 | Golden Principles + background auto-PR |

---

### Q3.5 Context Rot 是什么?怎么应对?

**答题要点**:
- **现象**:Anthropic 论文发现,上下文超过某阈值后,模型注意力衰减,表现断崖式下降。
- **应对**:
  1. **Context Reset**(Anthropic):**重启 > 修复**——与其让 agent 在污染的上下文里挣扎,不如清空重来。
  2. **主动压缩**:每 N 轮强制总结。
  3. **优先级排序**:把最近/最重要的放前面。

---

### Q3.6 Hermes Agent 和 OpenClaw 有什么区别?

**答题要点**:
- **Hermes**:目标**自进化**——agent 自己改自己的代码。
- **OpenClaw / PuinClaw**:目标**工具执行**——agent 用工具完成用户目标。
- 二者不矛盾,Hermes 是更激进的"自治代码库"方向。

---

## 4. 长跑与自治代码库(L3-L4)

### Q4.1 怎么让 Agent 长期跑?(几小时/几天)

**答题要点**(参考 Phase 9 笔记 04):

**5 个关键能力**:

| 能力 | 实现 |
| --- | --- |
| **Checkpoint** | 定期保存进度,中断可恢复 |
| **Cap** | turn 数 + cost USD 双限制 |
| **Pause/Resume** | 主动暂停 / 资源紧张暂停 |
| **Drift Detection** | 检测上下文污染 |
| **Auto-compact** | 主动总结历史 |

**3 种实现路线**:

| 路线 | 代表 | 特点 |
| --- | --- | --- |
| **简化实现** | pi-mom | while + 3 工具 + 双 cap |
| **Durable Memory** | Codex | 4 个 markdown 状态文件 |
| **极致简化** | Anthropic | bash `while true` + 16 并行 agent |

---

### Q4.2 pi-mom 的 goal runner 怎么实现?

**答题要点**:
- **核心循环**:`while(true)`,每轮检查 `aborted / signals.complete / signals.pause / turn cap / cost cap`。
- **3 个控制工具**:
  - `save_progress`——写 checkpoint。
  - `mark_complete`——主动标记完成,跳出循环。
  - `request_pause`——主动暂停,等用户介入。
- **默认配置**:`max_turns=100, max_cost_usd=$5, checkpoint_every=5`。

---

### Q4.3 Codex 的 durable project memory 是什么?

**答题要点**:
- **4 个 markdown 状态文件**:
  - `Prompt.md`——原始需求 + 变更记录。
  - `Plan.md`——当前计划 + 进度。
  - `Implement.md`——已做决策 + 待办。
  - `Documentation.md`——架构/约定。
- **优势**:markdown 可 git diff,跨会话连续,人类可读。

---

### Q4.4 Anthropic 的"16 个 Claude 写 C 编译器"是怎么跑的?

**答题要点**(参考 Phase 9 笔记 03/04):
- **Ralph-loop**:`bash while true` 包裹 agent。
- **16 并行 agent**:每个 agent 独立 git worktree。
- **Git lock 协调**:用 git lock 防止冲突。
- **整体特点**:工具层极简(就是 git + bash + Claude Code),靠**并行 + 循环**扛长跑。

**陷阱**:别以为这是"自治代码库"——它是 **Agent Teams 的极致案例**(多 agent 并行,不是单 agent 自治)。

---

### Q4.5 自治代码库(Autonomous Codebase)是什么概念?

**答题要点**:
- **4 个特征**:
  1. Agent 能**自己改仓库**(不只是写新代码)。
  2. Agent 能**自己定任务**(读 issue/PR/监控)。
  3. Agent 能**自己评审**(agent 给 agent 提 PR)。
  4. Agent 能**自己演进**(积累约定到 AGENTS.md)。
- **5 类漂移**(必须应对):

| 漂移 | 现象 |
| --- | --- |
| **目标漂移** | 跑偏了用户原意 |
| **质量漂移** | 越改越烂 |
| **风格漂移** | 代码风格混乱 |
| **依赖漂移** | 包版本失控 |
| **认知漂移** | AGENTS.md 越来越乱 |

---

## 5. Agent Teams(L3-L4)

### Q5.1 为什么要做多 Agent,而不是把所有能力塞进一个 Agent?

**答题要点**(小红书一面真题 #3,超高频):

**单 Agent 痛点**:
- 任务长、工具多、约束多时,**上下文污染**严重。
- 规划/检索/判断/执行混在一起,**错误放大**。
- 调试困难——不知道哪一步决策错。

**多 Agent 价值**:
- 把**角色分工显式化**(规划 Agent / 检索 Agent / 执行 Agent / 审查 Agent)。
- 每个 Agent 只负责一类清晰目标,**行为边界稳定**。

**3 个必要条件**(否则别上多 Agent):
1. 任务天然可分阶段。
2. 子任务之间职责边界清晰。
3. 单 Agent 已出现明显长链路退化。

**陷阱**:别说"多 Agent 看起来更高级"——多 Agent 引入**通信成本 + 状态同步复杂度 + 错误传播路径**,很多时候只是把单 Agent 的问题分散成多个。

---

### Q5.2 多 Agent 协作有哪些模式?

**答题要点**:

| 模式 | 特点 | 代表 |
| --- | --- | --- |
| **层级式(Hierarchical)** | 主 agent 调度子 agent | learn-claude-code Phase 5 Lead-Teammate |
| **平等式(Peer)** | agent 之间互相调用 | CrewAI |
| **竞争式(Competitive)** | 多 agent 投票/辩论 | Debate |
| **流水线(Pipeline)** | 顺序传递 | MetaGPT |

---

### Q5.3 多 Agent 之间怎么通信?

**答题要点**:
- **MessageBus 模式**(learn-claude-code Phase 5):
  - 所有 agent 注册到 bus。
  - 发消息 = 发布 topic。
  - 收消息 = 订阅 topic。
- **直接调用**:Lead 直接 spawn Teammate,等返回。
- **共享黑板(Blackboard)**:所有 agent 读写同一个状态文件。

**陷阱**:别只说"它们会互相沟通"——要说**具体通信机制**(同步/异步?消息格式?失败怎么办?)。

## 6. Agentic Sandbox(L3)

### Q6.1 怎么安全地让 Agent 跑代码?

**答题要点**(参考 Phase 9 笔记 05):

**4 层隔离**:

| 层 | 技术 | 隔离什么 |
| --- | --- | --- |
| **容器层** | Docker | 文件系统 + 进程 + 网络 |
| **OS 层** | namespace + cgroups | UID / PID / mount / 资源限额 |
| **语言层** | 沙箱解释器 | 系统调用过滤(seccomp) |
| **应用层** | 权限白名单 | 哪些 API 允许调用 |

---

### Q6.2 为什么不直接用 Docker,还要 namespace + cgroups?

**答题要点**:
- Docker **底层就是** namespace + cgroups 的封装。
- 但 Docker **隔离粒度粗**(整个容器),namespace 可以**细粒度到单进程**。
- Agent 场景下,启动一个容器跑代码 = 几秒;直接用 namespace = 毫秒级。
- **Codex 沙箱**:landlock + seccomp,不开 Docker,直接用 OS 原语。

---

### Q6.3 Sandbox 怎么防 prompt injection 触发的恶意代码?

**答题要点**:
- **网络白名单**:只允许特定域名(防止数据外泄)。
- **文件系统只读**:除工作目录外,全部 mount 为 read-only。
- **资源限额**:CPU / 内存 / 磁盘 / 时间 hard limit。
- **审计日志**:所有 syscall 记录,事后可追溯。

### Q7.2 DSPy 是什么?怎么用?

**答题要点**:
- **DSPy**:斯坦福开源,**提示词自动优化框架**。
- **核心思想**:把 prompt 当作**可优化参数**,用 examples 当训练集,自动调出最优 prompt。
- **流程**:
  1. 定义 task + metric。
  2. 收集 examples。
  3. 选 optimizer(如 `BootstrapFewShot`)。
  4. DSPy 自动调 prompt。
- **对比手动调**:更系统 / 可复现 / 不依赖灵感。

### Q7.3 怎么做 bad case 分析?

**答题要点**(小红书一面真题 #8 思路):

**bad case 4 分型**:

| 类型 | 原因 | 优化方向 |
| --- | --- | --- |
| **没召回到** | query rewrite / chunk / embedding | 改 chunk 策略 / 换 embedding |
| **召回没排上来** | rerank 问题 | 调 rerank 模型 / 融合策略 |
| **证据够生成错** | 模型推理 | few-shot / CoT |
| **证据冲突归因错** | 模型决策 | 加 verifier |

**关键**:线上闭环必须记录 query → 改写 → 召回 → 重排 → 上下文 → 答案全链路,否则只知道"用户不满意",不知道哪一步错。

### Q8.3 流式输出怎么实现?SSE 还是 WebSocket?

**答题要点**(花海真题·高频):
- **SSE(Server-Sent Events)**:
  - 单向(服务端→客户端)。
  - HTTP 协议,自动重连。
  - 适合 LLM 流式输出。
- **WebSocket**:
  - 双向。
  - 需要心跳保活。
  - 适合多用户交互场景。

**结论**:LLM 流式输出**默认用 SSE**(简单 + 自动重连),需要双向才上 WebSocket。

---

### Q8.4 Agent 调用工具超时怎么办?

**答题要点**:
- **指数退避重试**(3 次):1s / 2s / 4s。
- **熔断**:连续失败 N 次熔断 M 秒。
- **降级**:fallback 到更便宜的工具或返回"能力不可用"。
- **超时分级**:
  - 网络超时:重试。
  - LLM 推理超时:直接 abort(避免继续烧钱)。
  - 工具执行超时:kill 进程。

### Q8.5 怎么做 Agent 的可观测?

**答题要点**:
- **Trace**(每一步):思考 / 工具调用 / 参数 / 返回 / 耗时 / Token。
- **Metric**(聚合):成功率 / 平均步数 / 平均成本 / 工具使用频率。
- **Log**(审计):所有 LLM 输入输出 + 工具调用记录。
- **工具**:LangSmith / Langfuse / 自建 OpenTelemetry。
