# 04 - 综合:三层联动与数据流

> [!note]
> 本篇是把 [[01 - pi-agent-core]] + [[02 - pi-ai]] + [[03 - pi-coding-agent]] 串起来的综合笔记。
> 前三篇按"模块视角"看每一层,本篇按"运行视角"看**一个请求如何在三层之间流动**,以及每一段数据在不同层里以什么形态存在。
> 读完你应该能在脑子里画出:用户按一次回车之后,数据穿过哪些函数、哪些文件、哪些数据结构,最后变成屏幕上的一段文字。

---

## 1. 三层架构的"为什么"

pi-mono 不是随便切成三个包的。每一层解决一个**正交的问题**,层次边界就是**类型契约**:

```
┌──────────────────────────────────────────────────────────────────┐
│ coding-agent                                                     │
│   问题:用户怎么用?怎么管理会话/工具/扩展/凭证?                │
│   边界:AgentMessage(可被 declaration merging 扩展)             │
├──────────────────────────────────────────────────────────────────┤
│ agent-core                                                       │
│   问题:Agent 的循环怎么跑?工具怎么调度?事件怎么传?           │
│   边界:streamFn 函数契约 + AgentState 接口                      │
├──────────────────────────────────────────────────────────────────┤
│ ai                                                               │
│   问题:11 家 provider 怎么收敛成一套接口?SSE 怎么解析?         │
│   边界:StreamFunction / CompleteFunction + AssistantMessageEvent│
└──────────────────────────────────────────────────────────────────┘
```

**关键观察**:层与层之间**只通过类型契约耦合**,不通过具体实现。这意味着:
- 换 LLM(Anthropic → OpenAI)→ 只动 `ai` 层
- 换 runtime(双循环 → 单循环)→ 只动 `agent-core` 层
- 换产品形态(TUI → Telegram bot)→ 只动 `coding-agent` 层

这种切分不是过度设计 — 是真在 PiunClaw 改造里用得上的。`/goal` 长跑模式只动 `coding-agent`,`pi-agent-core` 和 `pi-ai` 一行不改。

## 2. 完整请求生命周期(一个 turn 的全流程)

这是本篇最核心的图。想象用户在 TUI 输入"修复 bug",按回车。下面是**一次 turn** 的完整数据流:

```
┌─────────────────────────────────────────────────────────────────────┐
│ USER INPUT                                                          │
│   "修复 bug"                                                        │
└─────────────────────────────────────────────────────────────────────┘
                  │
                  ▼
[coding-agent] AgentSession.prompt(userText)
                  │
                  ├─① 构建 UserMessage(带 timestamp/UUID)
                  ├─② append 到 agent.state.messages
                  ├─③ sessionManager.appendUserMessage(...) → 写入 session.jsonl
                  ├─④ emit("message_start") → UI/Modes 收到
                  └─⑤ 调 agent.prompt(userKick) ─────────────────┐
                                                                    │
                                                                    ▼
[agent-core] Agent 外层 followUp 循环                                │
   while (!shouldStop(state, event)) {                              │
      ┌────────────────────────────────────────────────────────┐    │
      │ ⑥ context = {                                          │    │
      │     systemPrompt: buildSystemPrompt(...),              │    │
      │     messages: state.messages,                          │    │
      │     tools: activeTools,                                │    │
      │     thinkingLevel, model, ...                          │    │
      │  }                                                     │    │
      └────────────────────────────────────────────────────────┘    │
                  │                                                  │
                  ▼                                                  │
   transformContext(messages)  ← extension hook: "context"           │
                  │                                                  │
                  ▼                                                  │
   convertToLlm(messages)     ← coding-agent 注入                   │
      作用:把 BashExecution/Custom/CompactionSummary                │
           这类扩展 message 转成 LLM 看得懂的 user/assistant/tool    │
                  │                                                  │
                  ▼                                                  │
   onPayload(payload)         ← extension hook:                      │
      "before_provider_request"                                      │
      作用:扩展可以最后改 systemPrompt / messages                   │
                  │                                                  │
                  ▼                                                  │
   streamFn(model, context, options) ──────────────────┐             │
                                                        │             │
                                                        ▼             │
[ai] streamSimple(model, context, options)                            │
   ① apiRegistry.getApi(model.provider) → 取 provider API            │
   ② wrapStream 包装                                                  │
   ③ api.stream(context, { apiKey, headers, ... })                   │
                  │                                                  │
                  ▼                                                  │
   provider/anthropic.ts (或其它)                                     │
      fetch("https://api.anthropic.com/v1/messages", {               │
        body: JSON.stringify({                                        │
          system: context.systemPrompt,                              │
          messages: <Message[]>,                                     │
          tools: <Tool[]>,                                           │
          ...                                                        │
        })                                                           │
      })                                                             │
                  │                                                  │
                  ▼                                                  │
   SSE chunks 流式返回                                                │
                  │                                                  │
                  ▼                                                  │
   parseStreamingJson SSE → 12 种 AssistantMessageEvent              │
      { type: "text", delta: "..." }                                 │
      { type: "thinking", delta: "..." }                             │
      { type: "tool_call", name, args }                              │
      { type: "tool_call_result", ... }                              │
      { type: "usage", ... }                                         │
      ... 共 12 种                                                    │
                  │                                                  │
                  ▼                                                  │
   yield event(给上层) ──────────────────────────────────┐           │
                                                          │           │
                                                          ▼           │
[agent-core] 收到事件流,内层 while 循环                              │
   for await (const event of stream) {                                │
      ┌──────────────────────────────────────────────────────┐       │
      │ ⑦ _state.listeners.emit(event)                       │       │
      │    → coding-agent 的 AgentSession 订阅                │       │
      │    → 持久化到 session.jsonl                           │       │
      │    → emit 给 UI/Mode                                  │       │
      ├──────────────────────────────────────────────────────┤       │
      │ ⑧ process(event):                                    │       │
      │    text → append 到 current assistant message        │       │
      │    tool_call → 收集到 pendingToolCalls               │       │
      │    usage → 累加到 state.totalUsage                    │       │
      │    stop → 退出内循环                                  │       │
      └──────────────────────────────────────────────────────┘       │
   }                                                                  │
                  │                                                  │
                  ▼                                                  │
   有 pendingToolCalls 吗?                                           │
      │                                                              │
      ├─ 没有 → 外层 followUp 循环判断要不要再跑                     │
      │                                                              │
      └─ 有 → 并行 or 串行执行每个 tool                              │
                  │                                                  │
                  ▼                                                  │
[coding-agent] ToolDefinition.execute(...)                           │
   ① prepare(args)        ← 给 LLM 一个 immediate feedback           │
   ② execute(args, signal)← 真正干活(read/write/bash/edit)          │
   ③ finalize(result)     ← 写日志/触发 extension hook                │
                  │                                                  │
                  ▼                                                  │
   tool_result 作为新的 message 入队                                 │
   跳回外层 while 循环,继续向 LLM 请求下一轮                        │
                                                                     │
   直到 mark_complete / pause / max_turns / max_cost                  │
                                                                     │
                  ┌──────────────────────────────────────────────────┘
                  ▼
[coding-agent] emit("message_end")
                  │
                  ▼
[mode/interactive] UI 渲染最终结果
```

**记住这 8 个关键点**:
1. AgentMessage(`coding-agent` 内部)→ Message(`agent-core`/`ai` 用)的转换在 `convertToLlm`
2. 系统提示词是**每轮重新构建**的(不缓存),所以 system-prompt.ts 的动态组装每轮都跑
3. extensions 有 3 个钩子点能改 prompt:`context` / `before_provider_request` / `convertToLlm` 间接
4. 工具是**外层循环里执行**,不在 LLM stream 里 — stream 只产出 `tool_call` 意图
5. 所有事件都走 `_state.listeners` — UI/Mode/Modes 都订阅同一份流
6. session.jsonl 是 append-only 的,失败不影响已写入
7. CompactionSummary / BranchSummary 也是 message,只是 role 不同
8. tool_call → execute → tool_result 这一整套是一个 turn 内的"子循环"

## 3. 数据在不同层的形态变化

同一段对话,在三层里以**不同的类型**存在。理解这个映射,就读懂了 pi-mono 的核心抽象。

### 3.1 用户消息的形态变化

```
[coding-agent 层]
   AgentMessage = UserMessage | AssistantMessage | ToolResultMessage
                | BashExecutionMessage       ← coding-agent 扩展
                | CustomMessage              ← coding-agent 扩展
                | BranchSummaryMessage       ← coding-agent 扩展
                | CompactionSummaryMessage   ← coding-agent 扩展
   ↓
[agent-core 层]
   convertToLlm() 把扩展 message "压扁" 成基础 message:
      bashExecution   → user (text)
      custom          → user (text/image)
      branchSummary   → user (text 带 prefix/suffix)
      compactionSummary → user (text 带 prefix/suffix)
   ↓
   Message = { role: "user"|"assistant"|"toolResult",
               content: TextContent[] | ImageContent[] | ToolCall[] | ...,
               timestamp }
   ↓
[ai 层]
   provider/anthropic.ts 把 Message 序列化成 Anthropic API 的 JSON:
      { role, content: [{type:"text",text:"..."}, ...] }
```

### 3.2 declaration merging 的妙处

`coding-agent/src/core/messages.ts` L70-77 用了 TypeScript 的 `declare module`:

```typescript
declare module "@mariozechner/pi-agent-core" {
    interface CustomAgentMessages {
        bashExecution: BashExecutionMessage;
        custom: CustomMessage;
        branchSummary: BranchSummaryMessage;
        compactionSummary: CompactionSummaryMessage;
    }
}
```

这意味着:`pi-agent-core` 的 `AgentMessage` 类型在 coding-agent 里**被扩展了**,但 agent-core 自己**完全不知道**这些类型存在。Compaction/分支/用户 bash 这些业务概念,在 agent-core 看来都是透明的 — 它只看到一个 message 数组,转给 streamFn 就完事。

这个设计让 agent-core 保持纯净(只关心循环 + 工具调度 + 事件),业务概念都在 coding-agent 里。**这是层次隔离的教科书案例**。

### 3.3 LLM 响应的形态变化

```
[Anthropic/OpenAI/... 网络层]
   SSE chunks:
      event: content_block_delta
      data: {"type":"text_delta","delta":"修复 "}
      data: {"type":"text_delta","delta":"这个 "}
      data: {"type":"text_delta","delta":"bug"}
      event: message_stop
   ↓
[ai 层]
   parseStreamingJson 把 raw SSE chunk 切成有意义的 JSON 片段
   ↓
   12 种 AssistantMessageEvent:
      TextEvent          { type:"text", delta:string }
      ThinkingEvent      { type:"thinking", delta:string }
      ToolCallEvent      { type:"tool_call", id, name, args }
      ToolCallResultEvent{ ... }
      UsageEvent         { type:"usage", input, output, cacheRead, cacheWrite }
      StopEvent          { type:"stop", reason:"end_turn"|"tool_use"|"..." }
      ErrorEvent         { type:"error", error:Error }
      ... 还有 5 种
   ↓
[agent-core 层]
   for await (const event of stream) {
      _state.listeners.emit(event);   // 给订阅者
      process(event);                 // 累积到 current assistant message
   }
   ↓
[coding-agent 层]
   AgentSession 订阅事件:
      ① 持久化到 session.jsonl
      ② emit("message_update", ...) 给 UI
      ③ 累积成完整的 AssistantMessage
```

**注意 12 种事件不是为了花哨,是因为不同 provider 的语义差异**:
- Anthropic 有 `thinking`(reasoning model),OpenAI 没有 → 事件里有 `ThinkingEvent` 但 OpenAI provider 永远不会发
- Gemini 有 `code` 直接生成,OpenAI 没有 → 同理
- 统一成 12 种,意味着 agent-core 不需要 if/else 区分 provider

## 4. AgentSession:把所有东西捏在一起

`agent-session.ts` 是 coding-agent 的**核心对象**(3289 行,文件最大)。所有 mode(interactive / print / rpc)都用同一个 AgentSession,只是 I/O 层不同。

### 4.1 它持有 7 个组件

```
AgentSession
   ├─ agent              : Agent          ← 来自 pi-agent-core 的 runtime
   ├─ sessionManager     : SessionManager ← 持久化 + 分支
   ├─ settingsManager    : SettingsManager← 用户偏好
   ├─ resourceLoader     : ResourceLoader ← AGENTS.md/skills/extensions 加载
   ├─ modelRegistry      : ModelRegistry  ← 已配置的模型
   ├─ extensionRunner    : ExtensionRunner← 钩子触发器
   └─ cwd                : string         ← 当前工作目录
```

### 4.2 它干的 6 类活

```
① 生命周期
   prompt(text)         → 一次 turn
   cancel()             → 中止当前请求
   fork()/switch()      → 分支/切换会话

② 模型管理
   setModel(model)
   setThinkingLevel(level)
   cycleModel(direction)

③ 工具管理
   setActiveTools(names)
   toggleTool(name)
   registerCustomTool(def)

④ Compaction
   compact(reason="manual"|"threshold"|"overflow")
   estimateContextTokens()

⑤ Bash 执行(用户 ! 命令,不是工具调用)
   executeBash(command)

⑥ 持久化
   每次 agent 事件 → sessionManager.append*(...)
   启动时 → sessionManager.buildSessionContext() 恢复
```

### 4.3 事件订阅:三层都共享同一份流

```
agent._state.listeners  ← agent-core 暴露的 EventEmitter
        │
        ├─ AgentSession 订阅 → 持久化 + 转发
        │     │
        │     └─ AgentSession 自己的 listeners(AgentSessionEvent)
        │           │
        │           ├─ interactive-mode 订阅 → 渲染 TUI
        │           ├─ print-mode 订阅      → 打印到 stdout
        │           └─ rpc-mode 订阅        → 序列化发 JSON-RPC
        │
        └─ extensions 订阅   → 触发钩子
```

**关键**:三个 mode 共用 AgentSession,差异只在最后一层 — "事件来了怎么显示"。这就是为什么 print-mode / rpc-mode / interactive-mode 各只有几百行代码,而 AgentSession 有 3289 行 — 业务全在 AgentSession 里。

### 4.4 AgentSessionEvent 扩展了 AgentEvent

```typescript
export type AgentSessionEvent =
   | AgentEvent                              // 继承 agent-core 的事件
   | { type: "queue_update"; steering; followUp }
   | { type: "compaction_start"; reason }
   | { type: "compaction_end"; result }
   | { type: "tool_execution_start"; ... }
   | { type: "tool_execution_end"; ... }
   | { type: "turn_start"; turn:number }
   | { type: "turn_end"; turn:number; usage }
   | ...
```

→ coding-agent 在 agent-core 的 12 种事件之上又加了 10+ 种业务事件。这些事件描述了"一次 turn 的开始/结束"、"工具执行"、"压缩"等业务概念 — 这些是 agent-core 不应该知道的。

## 5. 工具系统的完整执行细节

工具不只是 `execute()` — 它是一套**三件套 + 两个出口 + 两个模式**。这部分讲清楚,你就知道 pi-mono 的工具抽象有多完整。

### 5.1 三件套(prepare / execute / finalize)

```
ToolDefinition.execute() 内部分 3 步:

   ┌─────────────────────────────────────────────────────┐
   │ ① prepare(args)                                    │
   │   作用:生成 immediate outcome                       │
   │   场景:edit 工具收到 args 后,先算出 diff,        │
   │        立刻 yield 给 LLM 看 "我要做这个改动"      │
   │   注意:不真改文件                                  │
   └─────────────────────────────────────────────────────┘
                  │
                  ▼
   ┌─────────────────────────────────────────────────────┐
   │ ② execute(args, signal)                            │
   │   作用:真正干活                                    │
   │   场景:edit 真的写文件 / read 真的读 / bash 真的跑│
   │   signal:AbortSignal,用户中断时 reject            │
   └─────────────────────────────────────────────────────┘
                  │
                  ▼
   ┌─────────────────────────────────────────────────────┐
   │ ③ finalize(result)                                 │
   │   作用:收尾                                        │
   │   场景:记日志、触发 extension hook、清理临时资源  │
   └─────────────────────────────────────────────────────┘
```

### 5.2 两个出口(immediate vs prepared)

```
LLM 看到 tool_call → execute 后,有两个时机收到反馈:

出口 A:immediate outcome(prepare 阶段)
   edit 工具收到 {path, old_string, new_string}
      → prepare 算出 diff
      → 立刻 yield diff 给 LLM
      → LLM 看到 "我即将做这个改动" → 可以提前 abort
   
出口 B:prepared outcome(execute 完成后)
   execute 真的写入文件
      → 返回 final result(成功/失败 + 元数据)
      → LLM 看到 "已写入"

→ immediate 是预测,prepared 是事实
→ LLM 可以基于 immediate 决定要不要继续
```

### 5.3 两个模式(sequential vs parallel)

```
sequential(默认):
   tool_call_1 → execute → result_1
   tool_call_2 → execute → result_2
   tool_call_3 → execute → result_3
   
   一个一个跑。优点:文件状态可预测。
   缺点:慢。read x3 串行 = 3 倍延迟。

parallel:
   tool_call_1 ─┐
   tool_call_2 ─┼─→ Promise.all → [r1, r2, r3]
   tool_call_3 ─┘
   
   一起跑。优点:快。缺点:有副作用的工具(write/edit/bash)
   不能并行 — 会冲突。

判断规则(简化):
   有 file-mutation-queue 的工具(write/edit)→ 必须串行
   纯读工具(read/grep/find/ls)→ 可以并行
```

### 5.4 file-mutation-queue:串行化保障

`tools/file-mutation-queue.ts`(39 行)是个简洁的互斥锁:

```
write 工具:
   await fileMutationQueue.run(filePath, async () => {
      // 真的写文件
   });

edit 工具:
   await fileMutationQueue.run(filePath, async () => {
      // 真的改文件
   });

→ 同一文件的多次 write/edit 自动串行
→ 防止 edit A 改了一半,edit B 读到中间状态
```

这是 pi-mono 处理并发的核心 trick。简单,但关键。

## 6. Compaction:长会话的上下文治理

Compaction 是 pi-mono 处理 context 爆炸的机制。`compaction/compaction.ts`(715 行)是核心。

### 6.1 触发时机

```
shouldCompact(contextTokens, contextWindow, settings) (L219):

   ① contextWindow 的 80%(默认阈值)
      → 触发 "threshold" 类型 compaction
   
   ② provider 返回 context_overflow 错误
      → 触发 "overflow" 类型 compaction
      → compaction 完后重试请求
   
   ③ 用户手动 /compact
      → 触发 "manual" 类型 compaction
```

### 6.2 执行流程

```
compact(messages, model, ...) 内部:

   ① prepareCompaction(pathEntries, settings)  L612
      ─ 找到 cut point(在哪个 turn 边界切)
      ─ 区分 messagesToSummarize / turnPrefixMessages
      ─ 提取 fileOps(从历史里抽出文件操作)
      ─ 取 previousSummary(上次 compaction 的摘要)
   
   ② 触发 extension hook:"before_compact"
      ─ 扩展可以否决/修改 compaction 计划
   
   ③ generateSummary(messagesToSummarize, model, ...)  L530
      ─ 用当前 model 调一次 completeSimple
      ─ systemPrompt: SUMMARIZATION_SYSTEM_PROMPT
      ─ reasoning model → reasoning:"high"(让模型充分思考)
      ─ maxTokens = floor(0.8 * reserveTokens)
      ─ 如果有 previousSummary → 用 UPDATE_SUMMARIZATION_PROMPT(增量)
      ─ 否则用 SUMMARIZATION_PROMPT(首次)
   
   ④ 替换 messages:
      [CompactionSummaryMessage, ...turnPrefixMessages]
   
   ⑤ 写入 session.jsonl:
      新增 CompactionEntry { summary, tokensBefore, ... }
   
   ⑥ emit("compaction_end", { result })
```

### 6.3 CompactionSummaryMessage 的角色

```typescript
export interface CompactionSummaryMessage {
    role: "compactionSummary";   // ← coding-agent 扩展的 role
    summary: string;
    tokensBefore: number;
    timestamp: number;
}
```

这个 message 在 `convertToLlm` 里被翻译成:

```
"The conversation history before this point was compacted into the following summary:

<summary>
{summary text}
</summary>"
```

→ LLM 看到的是一段**普通 user message**,完全不知道这是压缩过的。这是抽象的边界 — provider/LLM 都不感知 compaction。

### 6.4 Branch Summary(分支摘要)

类似但有区别:

```
场景:用户 fork 一个分支去探索,探索完回到主线。
问题:探索分支的对话很长,合并回主线会污染主线 context。
解决:在 fork 点生成 BranchSummary,主线只看摘要,不看探索细节。

BranchSummaryMessage → 同样转成 user message,带 prefix/suffix 标签
```

→ Compaction 是"压缩自己",Branch Summary 是"压缩别人"。两者机制一样,语义不同。

## 7. Extension 钩子系统

extensions 不是简单的"插件",而是**贯穿整个生命周期的钩子链**。`extensions/types.ts` 定义了 20+ 个钩子点。

### 7.1 钩子点完整列表(按生命周期排序)

```
┌─────────────────────────────────────────────────────────────┐
│ 启动阶段                                                    │
│   resources_discover   发现 resources 时                    │
│   before_agent_start   Agent 第一次启动前                   │
├─────────────────────────────────────────────────────────────┤
│ 每轮 turn                                                   │

│ 错误处理                                                    │
│   error                任何 error 触发                     │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 钩子的执行模型

```typescript
// sdk.ts L307-313
onPayload: async (payload, _model) => {
    const runner = extensionRunnerRef.current;
    if (!runner?.hasHandlers("before_provider_request")) {
        return payload;                          // 快速路径:没钩子直接返回
    }
    return runner.emitBeforeProviderRequest(payload);
}
```

**关键设计**:
- 先检查 `hasHandlers` — 没注册就完全跳过(零开销)
- 多个扩展注册同一钩子 → 按 priority 顺序串行执行
- 钩子可以返回修改后的 payload(责任链模式)

### 7.3 ExtensionRunner 的状态管理

```
extensionRunnerRef: { current?: ExtensionRunner }
   ↑
   为什么用 ref 而不是直接持有?

因为:
   ① sdk.ts 创建 Agent 时,extensions 还没加载完
   ② ResourceLoader 异步 reload 后才有 extensions
   ③ 用 ref 让 Agent 持有"未来的"runner 引用
   ④ AgentSession 启动后:extensionRunnerRef.current = new ExtensionRunner(...)

→ 这是循环依赖的破解方法
```

## 8. 状态持久化的真实结构

session.jsonl 是 pi-mono 的会话存储。`session-manager.ts`(1419 行)管这个。

### 8.1 文件布局

```
~/.pi/agent/sessions/
   └─ <sessionId>/
      ├─ header.json          ← SessionHeader(sessionId/model/cwd/...)
      ├─ session.jsonl        ← append-only 事件流
      ├─ branches/
      │   ├─ main.jsonl       ← 主分支
      │   ├─ fork-abc123.jsonl← fork 的分支
      │   └─ ...
      └─ compactions/
         └─ <id>.json         ← 每次 compaction 的完整摘要
```

### 8.2 session.jsonl 的每行结构

每行是一个 JSON 对象,类型是 SessionEntry:

```typescript
type SessionEntry =
   | { type:"user_message"; id; timestamp; text }
   | { type:"assistant_message"; id; timestamp; content }
   | { type:"tool_call"; id; toolName; args; timestamp }
   | { type:"tool_result"; id; toolCallId; content; timestamp }
   | { type:"model_change"; fromProvider; fromId; toProvider; toId; timestamp }
   | { type:"thinking_level_change"; from; to; timestamp }
   | { type:"compaction"; id; summary; tokensBefore; timestamp }
   | { type:"branch_summary"; fromId; summary; timestamp }
   | { type:"branch_start"; branchId; parentId; timestamp }
   | { type:"branch_end"; branchId; timestamp }
   ...
```

### 8.3 恢复流程(sdk.ts L186-203)

```
buildSessionContext():
   ① 读 header.json → session 元数据
   ② 读 session.jsonl 按行解析 → SessionEntry[]
   ③ 按 entry 类型重建:
      - user_message    → UserMessage
      - assistant_message → AssistantMessage
      - tool_call/result → ToolCall/ToolResult
      - compaction      → CompactionSummaryMessage
      - branch_summary  → BranchSummaryMessage
      - model_change    → 恢复 model(注意:fallbackMessage)
      - thinking_level_change → 恢复 thinkingLevel
   ④ 返回 { messages, model, thinkingLevel }

→ 启动时 sdk 调用这个,得到完整上下文
→ agent.state.messages = existingSession.messages  (sdk.ts L329)
```

### 8.4 为什么用 jsonl 而不是 json

```
JSON 整文件:
   ✓ 一次性读完
   ✗ 任何 append 都要重写整个文件
   ✗ 写到一半 crash → 整个文件损坏

JSONL(每行一个 JSON):
   ✓ append 只写末尾(O(1))
   ✓ crash 损坏最后一行,前面完好
   ✓ 可以流式读取(大文件友好)
   ✗ 修改/删除某行需要重写
```

→ pi-mono 选择 JSONL 是因为 agent 会话**几乎只 append**(user/assistant/tool/compaction 都是追加)。这是**append-only 数据流的经典选择**。

## 9. 一次完整 turn 的伪代码(三层联动)

把前面的内容压缩成一段伪代码:

```typescript
// coding-agent/src/core/agent-session.ts
async prompt(text: string) {
    // ─── coding-agent 层 ───────────────────────────────
    const userMsg = createUserMessage(text);
    this.agent.state.messages.push(userMsg);
    this.sessionManager.appendUserMessage(userMsg);
    this.emit("message_start", { message: userMsg });
    
    // ─── 进入 agent-core 层 ────────────────────────────
    await this.agent.prompt({
        type: "user",
        text,
        source: "user",
    });
    // agent.prompt 内部跑外层 followUp 循环
}

// agent-core/src/agent.ts
async prompt(userKick) {
    while (!this.shouldStop()) {
        const context = {
            systemPrompt: this.systemPromptProvider(),
            messages: this.state.messages,
            tools: this.activeTools,
            model: this.state.model,
            thinkingLevel: this.state.thinkingLevel,
        };
        
        // 钩子:context
        context.messages = await this.transformContext(context.messages);
        
        // coding-agent 注入的转换
        const llmMessages = this.convertToLlm(context.messages);
        
        // 钩子:before_provider_request
        const payload = await this.onPayload({
            systemPrompt: context.systemPrompt,
            messages: llmMessages,
            tools: context.tools,
        });
        
        // ─── 进入 ai 层 ────────────────────────────────
        const stream = await this.streamFn(
            context.model,
            payload,
            { apiKey: await this.getApiKey() }
        );
        
        // 内层循环:消费事件流
        for await (const event of stream) {
            this.state.listeners.emit(event);        // 给订阅者
            
            if (event.type === "tool_call") {
                this.pendingToolCalls.push(event);
            } else if (event.type === "stop") {
                break;
            }
        }
        
        // 没工具调用 → 退出外层循环
        if (this.pendingToolCalls.length === 0) break;
        
        // 执行工具
        const results = await this.executeTools(this.pendingToolCalls);
        for (const result of results) {
            this.state.messages.push(result);       // tool_result 入队
            this.state.listeners.emit({ type: "tool_result", ... });
        }
        this.pendingToolCalls = [];
    }
}
```

**这段伪代码涵盖了三层**:
- `agent-session.prompt` 是 coding-agent 的入口
- `agent.prompt` 是 agent-core 的循环
- `streamFn → stream → events` 是 ai 层的产物

## 10. pi-mono 的设计哲学

读完三层 + 数据流,可以总结 pi-mono 的 5 条核心设计哲学:

### 10.1 类型契约取代运行时耦合

```
层与层之间不 import 对方的实现,只 import 类型。
   coding-agent → @mariozechner/pi-agent-core (类型 only)
   coding-agent → @mariozechner/pi-ai         (类型 only)
   agent-core   → @mariozechner/pi-ai         (类型 only)

实际行为通过函数注入:
   streamFn(model, context, options) → Stream
   convertToLlm(messages) → Message[]
```

### 10.2 declaration merging 让核心保持纯净

```
pi-agent-core 不知道 BashExecutionMessage / CompactionSummaryMessage。
它们通过 TypeScript declaration merging 加到 AgentMessage 联合类型。
pi-agent-core 只看到"有这些 role",但具体语义在 coding-agent。

→ 核心层稳定,业务扩展不污染核心
```

### 10.3 事件流是单一来源

```
所有状态变化都通过事件:
   agent._state.listeners.emit(event)
   ↓
   AgentSession.listeners.emit(sessionEvent)
   ↓
   Mode(UI)订阅 → 渲染

UI 永远不直接读 state — 只响应事件。
→ UI/Mode/AgentSession/Agent 四层完全解耦
```

### 10.4 工具是声明式的,自带元数据

```
ToolDefinition 不只是 execute()。它声明:
   - 如何介绍给 LLM(promptSnippet + promptGuidelines)
   - 如何在 TUI 显示(renderCall + renderResult)
   - 参数 schema(TypeBox)
   - 如何 prepare/finalize

→ system-prompt.ts / TUI / executor 都用同一个 ToolDefinition
→ 不需要在三处分别注册
```

### 10.5 append-only 持久化

```
session.jsonl 每行一个事件。
永不修改、永不删除(除非 compaction)。

→ crash 安全
→ 可以流式重放(recovery)
→ 分支/合并只是新增 entry
```
