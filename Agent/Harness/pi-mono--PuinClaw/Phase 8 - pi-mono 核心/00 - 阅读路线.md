## 1. 三包进度总览

```
┌─────────────────┬────────────────────────────────────┬──────────┐
│ package         │ 角色                               │ 状态     │
├─────────────────┼────────────────────────────────────┼──────────┤
│ agent           │ Agent runtime（loop + 状态机）     │ ✅ 完成  │
│ ai              │ LLM 调用层（streamFn + providers） │ ⏳ 进行  │
│ coding-agent    │ 真正的 coding CLI（工具+session）  │ ⬜ 待学  │
└─────────────────┴────────────────────────────────────┴──────────┘

依赖链(自底向上):
   ai ──→ agent ──→ coding-agent
```

**学习顺序**：agent（已完成）→ ai（当前）→ coding-agent（最后）

> [!tip] 为什么要按这个顺序
> agent 包定义了"运行时抽象"，但留了一个白：`streamFn` 到底是什么？
> ai 包就是回答这个白的。看完 ai，你能完整解释 agent 包里所有"留白"。
> coding-agent 才是产品层，要 agent + ai 都懂了才能看明白。

## 4. 重点问题（盯住这些）

### 4.1 ai 包的 4 个核心问题

| 问题 | 在哪里看 |
|---|---|
| 11 家 provider 怎么收敛成统一接口? | types.ts 的 `ApiProvider` + `stream.ts` |
| 新加 provider 怎么注册? | `api-registry.ts` + `register-builtins.ts` |
| LLM 流式响应怎么变成事件? | `anthropic.ts` 的 SSE 解析部分 |
| agent 包里的 `streamFn` 是什么? | `streamSimple` (stream.ts L43) |

### 4.2 跨包的核心认知（学完三包应能回答）

| 问题 | 答案在哪 |
|---|---|
| pi-mono 三层各自负责什么? | agent=runtime / ai=LLM适配 / coding-agent=产品 |
| 一个 prompt 进来后,数据怎么流到 LLM? | agent.ts → agent-loop.ts → stream.ts → provider |
| 事件怎么流回 UI? | provider → AssistantMessageEvent → loop → listeners |
| 怎么加一个新工具? | coding-agent 层定义 AgentTool,塞进 context.tools |
| 怎么换一个新 provider? | 实现 ApiProvider 接口 + registerApiProvider |
| 怎么拦截工具调用? | AgentLoopConfig.beforeToolCall / afterToolCall |
