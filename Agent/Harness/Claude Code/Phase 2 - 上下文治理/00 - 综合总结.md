# Phase 2 综合总结 --- 上下文治理

> [!note]
> Phase 1 把 Agent 跑起来了。但它有个隐性上限：**会话一长，模型就漂 / 爆 / 忘**。Phase 2 的三课（s05 TodoWrite、s06 Subagent、s08 Context Compact）合起来回答一个问题：**"在一个有限的上下文窗口里，怎么让 Agent 完成超过它容量的任务？"** 三种策略分别从"任务结构"、"上下文隔离"、"历史压缩"三个角度切入，互补而不重复。

## 为什么 Phase 2 是"上下文治理"

LLM 的上下文窗口是有限的（200K / 1M token）。但实际任务的需求是**无限的**：

- 一个跨多个文件的重构，需要看几十个文件，产生几十万 token。
- 一次代码调研，需要 grep 几十次、read 几十个文件，几万 token 起步。
- 一个跑了几十轮的会话，messages 列表本身就逼近上限。

如果什么都不做，messages 会**线性增长直到爆炸**。Phase 2 的三课是三种**非线性控制**手段：

| 课程 | 策略 | 控制什么 |
|---|---|---|
| s05 TodoWrite | **任务结构化** | 控制"模型把注意力放在哪里" |
| s06 Subagent | **上下文隔离** | 控制"哪些过程不进入主对话" |
| s08 Context Compact | **历史压缩** | 控制"老历史占多少空间" |

它们**叠加使用**：TodoWrite 让模型不漂、Subagent 让调研不污染、Compact 让历史不爆。任何一个单独用都不够。

## 每一步加了什么、为什么加

### s05 --- TodoWrite

| 维度 | 内容 |
|---|---|
| 加了什么 | `todo_write` 工具 + `CURRENT_TODOS` 全局状态 + SYSTEM prompt 引导 + nag 机制（3 轮没更新就提醒） |
| 为什么 | 长任务里模型注意力会被中间工具输出冲淡；自然语言"在心里计划"会在压缩中丢失；用户想看到进度 |
| 这是什么机制 | External Working Memory（外置工作记忆）；状态机（pending → in_progress → completed） |
| Claude Code 怎么做 | TaskCreate / TaskUpdate / TaskGet / TaskList 一组工具；支持 blockedBy 依赖、owner 字段；任务列表作为 system-reminder 定期重新注入 |

**关键贡献**：把模型的"短期记忆"从 messages 里抽出来，放到一个**不会被压缩、不会被工具输出冲淡**的位置。

**对应到 Claude Code**：Claude Code 的 task 系统更复杂（支持多 Agent 协作），但**核心机制完全一样**——用一个独立数据结构保存任务状态，循环里把它作为 reminder 注入。s05 是单 Agent 简化版。

### s06 --- Subagent

| 维度 | 内容 |
|---|---|
| 加了什么 | `task` 工具 + `spawn_subagent(description)` 函数；子 Agent 有独立 messages、独立 system、独立循环、30 轮上限；SUB_TOOLS 故意不含 task 防递归 |
| 为什么 | 探索型任务产生大量中间结果（grep 输出、文件内容），但这些结果对主 Agent 决策无用；隔离 = 可抛弃 |
| 这是什么机制 | Process Isolation / Fork-Join（和 Unix fork + pipe 同构） |
| Claude Code 怎么做 | 内置多种 subagent_type（Explore / Plan / general-purpose / code-reviewer）；支持 description 自包含指引；支持 run_in_background 并行；worktree 隔离（s18） |

**关键贡献**：确立了"委托 + 总结"模式。主 Agent 的上下文增长只等于"总结长度"，不是"探索过程长度"。

**对应到 Claude Code**：Claude Code 的 subagent 是产品化的——有专门化的角色、后台并行、worktree 隔离。但**核心契约一致**：description 自包含 + 返回总结。s06 是单层串行的简化版。

### s08 --- Context Compact

| 维度 | 内容 |
|---|---|
| 加了什么 | 四层渐进式压缩管线：L1 snip（条数裁剪）、L2 micro（旧 tool_result 占位）、L3 budget（大结果落盘）、L4 history（LLM 总结）；加反应式兜底 reactive_compact |
| 为什么 | 单一压缩策略（一刀切 LLM 总结）太贵太慢信息损失大；不同情况需要不同力度的压缩 |
| 这是什么机制 | Tiered Degradation / Progressive Compression（和 CPU L1/L2/L3 缓存、分代 GC 同构） |
| Claude Code 怎么做 | 自动压缩接近上限时触发；用户可见（CLI 显示 "Compressing context…"）；/compact 和 /clear 命令让用户主动控制；todo 列表被特殊保护不被压缩 |

**关键贡献**：定义了"压缩不变量"——todos、最近 N 轮、最初用户请求、SYSTEM prompt 永远不被压缩。

**对应到 Claude Code**：Claude Code 的压缩更精细（用户可见、可主动触发），但**分层思想完全一致**——先机械裁剪，再语义总结。s08 的四层管线是基础形态。

## 三种策略的关系

这三种策略不是"任选其一"，而是**叠加使用**：

```
              主对话上下文
    ┌──────────────────────────────────┐
    │                                  │
    │  ┌─────────────────────────┐     │
    │  │  todos (s05)            │     │  ← 任务结构化
    │  │  外置、不被压缩         │     │
    │  └─────────────────────────┘     │
    │                                  │
    │  messages: list                  │
    │  ┌────────────────────┐          │
    │  │ 早期消息           │          │
    │  │  ↓ snip (L1)       │          │  ← 历史压缩
    │  │  ↓ micro (L2)      │          │
    │  │  ↓ budget (L3)     │          │
    │  │  ↓ history (L4)    │          │
    │  └────────────────────┘          │
    │                                  │
    │  最近 N 轮 (不压缩)              │
    │                                  │
    └──────────────────────────────────┘
                ↑
                │ 总结返回
    ┌───────────┴────────────┐
    │  Subagent (s06)         │       ← 上下文隔离
    │  fresh messages[]       │
    │  探索过程在这里结束     │
    │  不进入主对话           │
    └─────────────────────────┘
```

每个策略的"分工"：

- **s05 TodoWrite**：解决"任务结构不清晰"。让模型有外置记忆，不靠脑力记任务。
- **s06 Subagent**：解决"探索过程污染主对话"。让探索性任务有独立空间。
- **s08 Context Compact**：解决"主对话本身被撑爆"。让历史占用可控。

**为什么需要三种**：

- 没 s05：模型在长任务里漂，不知道下一步做什么。
- 没 s06：一次调研就把主对话塞满，后续轮次都贵。
- 没 s08：会话一长就 API 报 prompt_too_long，Agent 死掉。

三者**机制不同、解决不同问题**，所以叠加有效。

## Phase 2 的统一逻辑：上下文经济学

把上下文看成一种**有限资源**（就像内存、磁盘），Phase 2 的三课是三种"管理资源"的工程实践：

| 资源管理策略 | 对应 Phase 2 |
|---|---|
| **减少分配**（少用一点） | s05 TodoWrite：模型不用脑力记任务，省出注意力 |
| **隔离分配**（独立地址空间） | s06 Subagent：探索过程在子上下文，不占主上下文 |
| **回收重利用**（GC / 分页） | s08 Context Compact：老历史压缩成总结 |

这和操作系统的内存管理是**完全同构**的：

- 进程隔离 ↔ Subagent
- 内存交换 ↔ Context Compact
- 进程内的栈/堆分工 ↔ TodoWrite

这种同构不是巧合。**任何"有限容量 + 多任务"的系统都会演化出类似机制**。LLM Agent 的特殊之处只在于：容量用 token 计量、回收靠 LLM 总结。

## 一个心智模型

想象你在写一本长篇小说：

- **s05 TodoWrite**：你贴一张大纲在墙上。每写完一章打勾。**这张大纲就是你的外置记忆**——你不需要把整本书的结构都装在脑子里。
- **s06 Subagent**：你雇一个研究助理帮你查资料。他给你一份**总结报告**，原始资料堆在他自己桌上（你不看）。
- **s08 Context Compact**：你写完 30 章后，把前 20 章的草稿**整理成一份"前情提要"**，把原文收进抽屉。你桌上只放前情提要 + 最近 10 章 + 大纲。

三个动作的目的都是同一个：**让你眼前这张桌子（上下文）只放你现在最需要的东西**。桌子大小不变，但你永远能专注当下。

## Phase 2 之后能做什么

到 s08 为止，harness 已经能：

- 跑跨几十轮的长任务（s05 防漂）
- 跑探索性任务（s06 隔离）
- 跑超过原始上下文容量的任务（s08 压缩）

**这是一个能跑长任务的 Agent**。它可以重构一个中型代码库、做一次完整的代码审查、写一份跨多个文件的设计文档。

但它还**不能**：

- 跨会话记住用户偏好（每次重启都从头开始）→ Phase 3 s09 Memory
- 自定义 system prompt（注入项目规则、用户偏好）→ Phase 3 s10 System Prompt
- 从错误中恢复（一次失败就死）→ Phase 3 s11 Error Recovery
- 跑后台、定时任务（必须用户在场）→ Phase 4 s12 - s14

Phase 3 解决"跨会话的连续性"。Phase 4 解决"无人在场的长任务"。

## 实现对照：Phase 2 之后的 agent_loop

```python
rounds_since_todo = 0

def agent_loop(messages: list):
    global rounds_since_todo
    while True:
        # s08: 四层压缩管线（循环开头跑）
        tool_result_budget(messages)   # L3
        snip_compact(messages)         # L1
        micro_compact(messages)        # L2
        if estimate_tokens(messages) > CONTEXT_LIMIT:
            compact_history(messages)  # L4

        # s05: nag reminder
        if rounds_since_todo >= 3 and messages:
            messages.append({"role": "user",
                             "content": "<reminder>Update your todos.</reminder>"})
            rounds_since_todo = 0

        try:
            response = client.messages.create(
                model=MODEL, system=SYSTEM, messages=messages,
                tools=TOOLS, max_tokens=8000,
            )
        except APIError as e:
            # s08: 反应式兜底
            if "prompt_too_long" in str(e):
                reactive_compact(messages)
                continue
            raise

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            force = trigger_hooks("Stop", messages)
            if force:
                messages.append({"role": "user", "content": force})
                continue
            return

        rounds_since_todo += 1
        results = []
        for block in response.content:
            if block.type != "tool_use":
                continue

            blocked = trigger_hooks("PreToolUse", block)
            if blocked:
                results.append({"type": "tool_result",
                                 "tool_use_id": block.id,
                                 "content": str(blocked)})
                continue

            handler = TOOL_HANDLERS.get(block.name)
            output = handler(**block.input) if handler else f"Unknown: {block.name}"

            trigger_hooks("PostToolUse", block, output)

            if block.name == "todo_write":
                rounds_since_todo = 0  # s05: 重置 nag 计数

            results.append({"type": "tool_result",
                             "tool_use_id": block.id, "content": output})

        messages.append({"role": "user", "content": results})
```

注意 Phase 2 相对 Phase 1 加的东西**非常少**：

- 循环开头多了 4 行压缩管线（s08）。
- 循环开头多了 3 行 nag reminder（s05）。
- try/except 包了一圈 API 调用（s08 反应式）。
- 循环体里多了"todo_write 时重置计数"（s05）。

**循环骨架本身没动**。s06 的 subagent 是通过加一个 `task` 工具 + `TOOL_HANDLERS["task"] = spawn_subagent` 接入的，完全在循环之外。

这就是 Phase 1 强调的"加而不破"原则的回报：到 Phase 2 为止，所有扩展都没改循环骨架。

## Q&A

### Q1: 为什么 s05 不放在 Phase 1？它看起来是个普通工具。

**A**: 因为 s05 解决的不是"Agent 能跑"的问题（Phase 1 的目标），而是"Agent 跑长"的问题（Phase 2 的目标）。

Phase 1 的 Agent 没有 todo_write 也能干活——只是任务一长就漂。s05 的存在动机是"长任务的上下文治理"，所以归在 Phase 2。

判断一个扩展属于哪个 Phase，**看它解决什么问题**，不看它用什么机制。

### Q2: s07（Skill Loading）为什么跳过？

**A**: 因为 s07 是"知识注入"机制，不是"上下文治理"机制。它和 Phase 2 的主题（治理已有上下文）不同。

s07 处理的是"如何把外部知识（文档、教程、规范）按需注入"——属于 Phase 6（扩展与组装），和 MCP（s19）同类。

Phase 编号不是严格按教程原顺序，而是按**逻辑归类**。原教程作者把 s07 放在第 7 位，是因为他想讲完 subagent 之后立刻讲知识注入。但从概念上，s07 更接近"外部资源接入"主题。

### Q3: Subagent 和 Context Compact 不都是"省上下文"吗？为什么不算一个？

**A**: 它们省的角度不同：

- **Subagent**：**主动**省——预期某个任务会爆，提前把它隔离。
- **Compact**：**被动**省——上下文已经满了，事后压缩。

类比：

- Subagent 像"你提前把这事外包出去，自己不操心"。
- Compact 像"你桌子满了，必须整理一下"。

工程上两者**互补**：你既希望尽量用 Subagent 隔离（提前省），也得有 Compact 兜底（防止意外爆）。任何长跑的 Agent 两个都要有。

### Q4: TodoWrite 看起来很弱（就一个 list），为什么 Claude Code 也用？

**A**: 因为它**便宜且有效**。

便宜：todo 是个 list 数据结构，没有 IO，没有 LLM 调用，token 占用极小。

有效：模型在长任务里漂是一个**事实**，不是错觉。任何"让模型重新看到任务列表"的机制都能显著改善。s05 的 nag reminder 已经能解决大部分漂移问题。

Claude Code 的 task 系统复杂（依赖、owner、多 Agent 协作），那是因为它要服务**多 Agent 场景**。在单 Agent 场景下，s05 的极简 list 已经够用。

**工程经验**：先用最简版本（s05），等真不够用再升级（Claude Code 的 task 系统）。

### Q5: Context Compact 会不会丢重要信息？

**A**: **会**。这是压缩的本质代价。

但 s08 的设计**最小化这个风险**：

1. **L1 / L2 / L3 是机械式**：只丢"明显不重要"的（旧 tool_result、中间消息）。这些信息即使丢了，模型大概率已经基于它们做了决定。
2. **L4 是语义式**：让 LLM 自己判断什么是重要的。LLM 的判断通常比硬规则准。
3. **保护"不变量"**：todos、最近 N 轮、最初请求永远不压缩。

但即使这样，长任务跑了几十小时后，模型可能"忘了"早期某个关键细节。这时候**用户要主动补**：要么说一遍"我之前提过 X"，要么用 todo 把它写成永久外置记忆。

**Compaction is lossy by design**。设计目标不是"零损失"，而是"可控损失"——丢掉不重要的，保留重要的。

### Q6: Phase 2 之后，我已经能写一个"产品级 Agent"了吗？

**A**: **还不能**。Phase 1 + 2 给你的是"能跑长任务的脚本"。要变成产品还需要：

- **持久化记忆**（Phase 3 s09）：跨会话不丢用户偏好。
- **错误恢复**（Phase 3 s11）：API 失败、工具失败不能让 Agent 死。
- **后台 / 定时**（Phase 4）：用户不在场也能跑。
- **多 Agent 协作**（Phase 5）：复杂任务拆给多个角色。
- **生态接入**（Phase 6 s19 MCP）：连接外部工具和数据源。

Phase 2 解决的是"会话内能跑长"。Phase 3 - 6 解决"会话外也能跑长"。
