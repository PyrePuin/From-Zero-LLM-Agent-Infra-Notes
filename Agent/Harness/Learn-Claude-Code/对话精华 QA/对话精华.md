# 对话精华 Q&A

>
> **同步规则**：每个 s 笔记底部都有自己的 Q&A 节（局部视角），本文件是**跨笔记的索引和精华**——保留最关键的几条 + `[[]]` 链回原笔记。

## 消息格式与协议

### Q1: message 的 content 到底是什么结构？一条消息里能有多个 block 吗？

**A**: 一条 message 的 `content` 有两种合法形态：

1. **字符串**：`"content": "看一下当前目录"`。最简单的用户输入或文本回复。
2. **block 列表**：`"content": [{type: "text", ...}, {type: "tool_use", ...}, ...]`。

**一条 assistant 消息完全可以同时包含多个 block**——比如先输出一段思考文字，然后并发调用 3 个工具，全在一条消息里。协议层面是合法的，模型经常这么干。

工具结果必须是 `role: "user"` + content 是 `[{"type": "tool_result", "tool_use_id": "...", "content": "..."}]`。这是协议规定的——工具结果属于用户方。

### Q2: tool_use 和 tool_result 怎么配对？

**A**: 通过 `id` 字段配对，**不是靠顺序**。模型在一条消息里并发出 5 个 tool_use（每个有唯一 id），harness 执行完把 5 个 tool_result 用同样的 id 回填，顺序可以变。模型按 id 匹配。

这意味着循环里**不能漏 tool_result**——少回一个，API 直接报错"missing tool_use_id"。

### Q3: s07 的 skill 如果 SKILL.md 之外还有 scripts 文件夹，模型怎么调用？

**A**: SKILL.md 是**manifest（清单）**，不是可执行入口。它告诉模型"这个 skill 干什么、什么时候用"。具体动作（包括跑脚本）**仍然通过现有的 bash / read_file / write_file 工具完成**。

流程是：

1. SYSTEM prompt 里有 skill 目录（名字 + 一句话描述）。
2. 模型判断需要某个 skill → 调 `load_skill(name)` 工具。
3. load_skill 返回完整 SKILL.md 内容。
4. 模型读完 SKILL.md，按里面的指引**调用 bash 跑脚本**（或者别的工具）。

没有"专门跑 skill 脚本的工具"。skill 是**知识注入**，不是新工具。

## Python 实现细节

### Q4: s08 的 micro_compact 里这段代码为什么生效？

```python
for block in msg["content"]:
    if block.get("type") == "tool_result":
        block["content"] = "(prior tool output elided)"
```

为什么改 block 的字段，messages 里就跟着变了？像 C 语言的指针？

**A**: 是的，这是 Python 的**引用语义**，和 C 指针概念上很像。

- Python 里 dict 是**堆上的对象**，变量名是**指向这个对象的引用**（指针）。
- `for block in msg["content"]` 里，`block` 是指向列表元素的引用。
- `block["content"] = "..."` 改的是**堆上 dict 的字段**，所有指向这个 dict 的引用（包括 `msg["content"][i]`）都看到新值。

但**重新赋值不一样**：

```python
block = {"type": "tool_result", "content": "..."}  # 只改局部变量
# 原 dict 不变，msg["content"][i] 还是旧值
```

`block = ...` 只改了局部变量指向哪个对象，没动原对象。所以必须**改字段**而不是**换对象**。

类似的，主循环的 `history` 持有 messages list 的引用。compact_history 内部 `messages = [...]` 只改局部引用，外部 history 不变；`messages.clear(); messages.extend(...)` 改的是堆上 list 的内容，外部 history 才看到。

### Q5: 为什么 Agent loop 每轮都要传 `tools=TOOLS`？

**A**: Anthropic API 是**无状态的**——服务端不保存会话状态，每次调用都是独立的。所以每轮都要把：

- system prompt
- 完整 messages 历史
- 可用工具列表

全发一遍。服务端按这些输入算输出，不记得你上一轮传过什么。

## 子 Agent 与记忆

### Q6: 子 Agent 能看到主 Agent 的对话吗？

**A**: **看不到**。spawn_subagent 里 messages 是**全新的** `[{"role": "user", "content": description}]`。主 Agent 的历史完全不存在于子 Agent 的视野。

这是 subagent 隔离的本质——内存里不存就影响不到。代价是**主 Agent 必须在 description 里把所有必要上下文传过去**。description 必须自包含。

### Q7: extract_memories 只看最近 10 条消息能提取出用户偏好吗？

**A**: 不是"会话级"的提取，是**每轮"用户提问 → Agent 完成"循环都尝试提取一次**。

具体策略：

- 每次会话有"提取机会"（一般是 stop 时），看**最近 10 条消息**这个**滑动窗口**。
- 窗口够小（10 条）以保证提取本身不消耗太多 token。
- 窗口够大（10 条）以覆盖"用户提需求 → Agent 跑完"的典型一轮。
- 多次会话 / 多轮触发累计起来，长期偏好自然会被多次提到、被反复写入并 consolidate。

这是**增量提取 + 长期累积**的策略。单次提取可能没价值，但跑多了就有。代价是会反复触发 LLM 调用（贵）。生产实现会做去重、合并、consolidate。

### Q8: memory_turn 有什么作用？为什么必须提前记录？

**A**: 因为**注入记忆会污染原 messages**，必须靠 memory_turn 知道"哪一段是注入的"。

具体场景：

1. 进入 agent_loop，调用 `load_memories(messages)` 拿到相关记忆。
2. 把记忆**注入到 user 消息**（追加一段 `<memories>...</memories>`）。
3. Agent 跑完，触发 `extract_memories(messages)`。

问题：如果直接在原 messages 里注入，extract_memories 会**看到注入的记忆内容**，可能把它们当成"用户说过的"再次提取——形成"提取 → 注入 → 再提取"的死循环。

解决：

- **不在原 messages 注入，而是注入到副本**。
- 记录 `memory_turn`：标记"这一轮是注入了记忆的"。
- 提取时**跳过 memory_turn**，只看原始的用户消息。

所以 memory_turn 存在的本质原因是：**我们选择用副本注入（保原 messages 干净），所以需要外部记录"哪一轮有注入"**。如果改成"原地注入"，就不需要 memory_turn，但会失去原 messages 的纯净性，影响后续处理。

设计选择：**用一点外部状态换取原数据的完整性**。

## 概念辨析

### Q9: Agent Loop 和 ReAct 是一回事吗？

**A**: **概念上同源，实现上是同一个东西的不同名字**。

- **ReAct**（Reasoning + Acting）是 2022 年论文提出的**框架/范式**：思考 → 行动 → 观察 → 再思考。
- **Agent Loop** 是这个范式在工程上的**实现**：while 循环 + tool dispatch + result feedback。

s01 的 agent_loop 就是 ReAct 的最简实现。Anthropic 的 tool_use API 是为这个范式设计的——`stop_reason == "tool_use"` 就是"现在该 Act 了"的信号。

### Q10: hooks 和 middleware（Express / Django）什么关系？

**A**: **同一个模式的两种叫法**。

- **Middleware**（中间件）：Web 框架里的常见叫法，强调"链式处理请求 / 响应"。
- **Hook**（钩子）：通用叫法，强调"在某个时机触发"。

核心要素一致：

- 注册多个回调。
- 按顺序执行。
- 任何一个可以短路（拒绝请求 / 拒绝执行）。

Claude Code 的 PreToolUse hook 和 Express 的 `app.use((req, res, next) => ...)` 本质相同——都在"主逻辑之前"插一段自定义代码，能拒绝就拒绝，不能就 next()。

### Q11: 上下文压缩和"丢弃中间消息"有什么区别？

**A**: **前者包含后者，但语义不同**。

- **丢弃中间消息**（L1 snip_compact）：纯机械——超过 N 条就删中间。不关心内容。
- **上下文压缩**（完整概念）：分层管线，包括机械裁剪（L1 - L3）+ 语义总结（L4）。

L4 才是真正意义上的"压缩"——它**重新表达**历史（用更短的总结替换更长的事实），保留语义。前 3 层更准确的叫法是"裁剪"（trimming）。

但日常交流里"上下文压缩"常指整个管线。

### Q12: subagent 和"开新会话"有什么区别？

**A**: **目的和返回方式不同**。

| 维度 | subagent | 开新会话 |
|---|---|---|
| 触发 | 主 Agent 主动 | 用户主动 |
| 上下文 | 干净，但 description 自包含 | 完全空白 |
| 结果 | 返回总结给主 Agent | 用户自己看 |
| 目的 | 主 Agent 委托子任务 | 用户切话题 |

subagent 是**主 Agent 的延伸**，开新会话是**用户的中断**。

### Q13: TodoWrite 是给用户的还是给模型的？

**A**: **首要给模型**，其次给用户。

模型用它做**外置工作记忆**——记住任务列表、跟踪进度、防止漂移。这是设计 todo 的核心目的。

渲染给用户看是**附带价值**——用户体验更好、能监控方向。但即使不渲染，模型自己也从中受益（每个 todo 都作为 tool_result 回到 messages，模型在后续轮次能看到自己列的任务）。

所以**不要给 todo 加模型用不到的字段**（deadline、assignee、优先级）。每个字段都消耗 token，模型用不上的就是浪费。

## 学习方法

### Q14: 学 s01 - s08 之后最大的收获应该是什么？

**A**: **能在脑子里画出 Agent 的结构图**：

```
        ┌─────────────────────────────────┐
        │           Agent Loop             │
        │   while stop == tool_use:        │
        │     1. PreToolUse hooks          │
        │     2. dispatch (TOOL_HANDLERS)  │
        │     3. PostToolUse hooks         │
        │     4. feed result back          │
        └─────────────────────────────────┘
                       ↑
        ┌──────────────┴───────────────┐
        │                              │
   ┌────┴─────┐                  ┌─────┴────┐
   │ messages │ ←── compact ──── │  memory  │
   │  list    │      (s08)       │  (s09)   │
   └────┬─────┘                  └──────────┘
        │
   ┌────┴─────┐
   │  todos   │ ←── model writes ──┐
   │  (s05)   │ ── rendered ──→ user
   └──────────┘
        │
   ┌────┴──────────┐
   │   subagent    │  (s06)
   │ fresh messages│
   └───────────────┘
```

每一块的存在都能用"为什么需要它"回答。然后看真正的 Claude Code（或任何 Agent 框架），能对应出每一块在哪个文件、加了哪些 s01 - s08 之外的扩展。

### Q15: 后续 s09 - s20 大致怎么演化？

**A**: 沿着三条线：

1. **更长的记忆**：s09 memory（跨会话）、s10 system prompt（动态注入）、s11 error recovery（容错）。
2. **更长的任务**：s12 task system（任务抽象）、s13 background（后台）、s14 cron（定时）。
3. **更多的 Agent**：s15 teams、s16 protocols、s17 autonomous、s18 worktree（多 Agent 隔离）。
4. **更广的生态**：s07 skill、s19 MCP（外部工具接入）、s20 综合。

骨架不变（[[01 - Agent Loop]] 的循环），扩展方向都是"让 Agent 处理更复杂、更长、更多协作的场景"。
