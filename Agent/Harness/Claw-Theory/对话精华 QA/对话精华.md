## 通道与并发（s04 Channels）

### Q1: `InboundMessage` 是 channel→agent 还是 agent→channel？

**A**: **channel → agent**。"Inbound" 站在 agent 视角——进入 agent 大脑的消息。出口方向（agent 回复）没有 `OutboundMessage` 类型，就是普通 `str` 传给 `channel.send(to, text)`。归一化成本只花在入站（入站结构复杂、出站简单）。

详见 [[04 - Channels#Q1]]。

### Q2: `run_agent_turn` 里 `messages.append({"content": inbound.text})` 看起来没把 Telegram 输入加进去？

**A**: `inbound` **就是** Telegram 消息——主循环 `for m in tg_msgs: run_agent_turn(m, ...)` 把后台线程解析好的 `InboundMessage` 传进来。`inbound.text` 已经是用户文字。到了这层，Telegram/CLI/飞书完全等价，这就是归一化的意义。

详见 [[04 - Channels#Q2]]。

### Q3: `ChannelManager.accounts` 和 `channels` 有什么区别？

**A**: 职责分离：

- `channels: dict[str, Channel]` —— **能干什么**（活的 receive/send 实例）
- `accounts: list[ChannelAccount]` —— **配了什么**（token / app_id 等配置记录）

`accounts` 在 s04 唯一用途是 `/accounts` REPL 命令打印 bot 列表。分开是为了未来支持"动态重载配置"。详见 [[04 - Channels#Q3]]。

### Q4: `run_agent_turn` 的 `else` 分支处理什么情况？

**A**: 兜底 `stop_reason` 不是 `end_turn` 也不是 `tool_use` 的情况——`max_tokens`（截断）、`stop_sequence`（停止词）、`pause_turn`（暂停）。模型可能已吐出部分文本但没说完，把这些片段发给用户然后 break（没法继续循环）。详见 [[04 - Channels#Q4]]。

### Q5: `select.select([sys.stdin], [], [], 0.5)` 是干什么的？

**A**: **非阻塞读 stdin** 的 POSIX 标准写法——把 stdin 当文件描述符 poll，最多等 0.5 秒，超时返回空列表。让主循环在"等 stdin 输入"和"排空 Telegram 后台队列"之间轮转。如果用阻塞 `input()`，主线程卡住，Telegram 队列堆着没人处理。详见 [[04 - Channels#Q6]]。

### Q6: s04 为什么要现场构造 `InboundMessage(text=user_input, channel="cli", ...)`？

**A**: 开了 Telegram 时主循环走非阻塞分支（`select`），只拿到裸字符串 `user_input`。手动包装成 `channel="cli"` 的 InboundMessage，让它走和 Telegram 完全相同的入口。没开 Telegram 时走 `cli.receive()` 直接返回 InboundMessage，就不需要这步。详见 [[04 - Channels#Q8]]。

### Q7: `for m in tg_msgs: run_agent_turn(m, ...)` 的串行限制是什么？

**A**: `run_agent_turn` 同步阻塞——10 条 Telegram 消息要串行处理 10 次，最后一条要等前 9 条处理完。这是 s04 的限制，**s10 Concurrency** 用"命名 lane"解决：不同 `peer_id` 走不同 lane 并行，同一 `peer_id` 仍串行（保证对话顺序）。详见 [[04 - Channels#Q9]]。

### Q8: 学 claw0 s04 要重点看哪几个函数？

**A**: 只看 6 个抽象层函数（`InboundMessage` / `build_session_key` / `Channel` ABC / `ChannelManager` / `run_agent_turn` / `agent_loop`）。`TelegramChannel` / `FeishuChannel` 内部的所有 `_buf_*`、`_flush_*`、`_parse`、`_refresh_token` 都是平台细节，**用到时再查**。真正的价值在 s05+（Gateway/Heartbeat/Delivery 这些通道无关架构）。详见 [[04 - Channels#Q11]]。

### Q9: claw0 的讲解是不是比 learn-claude-code 简化？

**A**: 数据反过来——claw0 单节均 doc 180 行 + code 747 行，learn-claude-code 是 115 + 330。差异在**风格**：claw0 是"参考手册"（是什么/怎么做），learn-claude-code 是"教程"（为什么/演进）。读 claw0 官方 md 时"为什么"部分得自己补。详见 [[04 - Channels#Q13]]。

## 路由与会话隔离（s05 Gateway & Routing）

### Q10: s05 是把多 agent 与"消息来源"绑定吗？

**A**: 不只是"消息来源"（channel）——那是 5 个匹配维度之一（T4）。更准确说 **"按 5 个粒度维度把消息路由到 agent"**：T1 peer_id / T2 guild_id / T3 account_id / T4 channel / T5 default。只用 channel 会强制"Telegram 来的消息全部归同一个 agent"，无法表达"特定用户专属"。详见 [[05 - Gateway & Routing#Q2]]。

### Q11: s05 的 AgentManager._sessions 真的落盘了吗？

**A**: **没有**。`_sessions` 是纯内存 dict，进程崩全丢。`register()` 里建的 `agent_dir / "sessions"` 目录是**架构预留**——告诉读者"未来这里会存 session 文件"，s05 自己不实现。整个 claw0 s01-s10 里只有 s03 实现了完整 session 持久化。详见 [[05 - Gateway & Routing#Q3]]。

### Q12: 为什么 `dm_scope` 挂在 AgentConfig 而不是 Binding 上？

**A**: 同一 agent 在不同 binding 下行为应一致（Luna 永远 `per-peer`）。配置聚合：性格 + 模型 + 隔离粒度是一组连贯属性。简化路由表：Binding 只管"消息归谁"，不管"怎么存对话"。详见 [[05 - Gateway & Routing#Q5]]。

### Q13: s05 为什么用 threading + asyncio 混用？

**A**: 历史包袱 + 渐进式教学。s04 是纯 threading（TG 后台轮询），s05 想引入 `Semaphore(4)` 并发限流又不重写 s04，于是 daemon 线程跑 asyncio 循环，跨线程用 `run_coroutine_threadsafe` 桥接。OpenClaw 生产代码用纯 asyncio 避免混用的死锁和调试困难。详见 [[05 - Gateway & Routing#Q6]]。

### Q14: 学 s05 要重点看哪几个函数？

**A**: 必读 5 个抽象层（Binding + BindingTable + build_session_key + AgentConfig + resolve_route，共 ~110 行）。跳过 GatewayServer（113 行 WebSocket 协议样板）和 REPL 命令实现。**只看抽象层 + 串联，s05 就过关了**。详见 [[05 - Gateway & Routing#Q8]]。

## 智能层（s06 Intelligence）

### Q15: 为什么 `build_system_prompt` 每轮都重拼，不缓存？

**A**: 因为 memory 可能上一轮刚被写入（模型调了 `memory_write`），下一轮 `_auto_recall` 会搜到这条新记忆。缓存 prompt 会漏掉新记忆。s06 选择"宁可重拼也不漏"。代价低（字符串拼接是微秒级 CPU）。详见 [[06 - Intelligence#Q1]]。

### Q16: `_auto_recall` 每条用户消息都跑，性能扛得住吗？

**A**: 扛得住。算法搜索（TF-IDF + 哈希投影 + MMR）是纯 Python 字典运算，1000 条记忆也是毫秒级。对比 learn-claude-code [[09 - Memory|s09]] 用 LLM side-query（~$0.01 + 几百毫秒每次），s06 的纯算法**便宜 1000 倍**——这就是 s06 敢每轮搜、s09 只敢 agent_loop 进来时搜一次的根本原因。详见 [[06 - Intelligence#Q2]]。

### Q17: `bootstrap_data["MEMORY.md"]` 和 `_auto_recall` 搜出来的记忆有什么区别？

**A**: 都进第 5 层，但语义不同：

- `bootstrap_data["MEMORY.md"]`：启动时一次性加载**整篇** MEMORY.md（受 20k 截断），进 `### Evergreen Memory`。**永远在，不挑相关性**。
- `_auto_recall` 结果：每轮实时搜的**片段**（MEMORY.md 段落 + JSONL 条目），进 `### Recalled Memories`。**按相关性挑**。

类比：前者是"档案"，后者是"针对当前问题的检索结果"。详见 [[06 - Intelligence#Q3]]。

### Q18: s06 和 learn-claude-code s10（System Prompt）的本质差异是什么？

**A**: 4 维差异：

| 维度 | s10 | s06 |
|---|---|---|
| 数据源 | sections 写死在代码里 | 从 `workspace/*.md` 读 |
| 组装 | 按 context **动态选** section | 固定 8 层顺序，每轮全拼 |
| Memory 层 | 无 | 第 5 层：每轮 `_auto_recall` |
| 多 agent | 单 agent | 每 agent 独立 workspace（s05 才真用上） |

**核心抽象**：s10 是"按状态选 section"的智能组装，s06 是"从磁盘 + 记忆实时组装"的数据驱动组装。"磁盘驱动"才是 s06 的灵魂，多 agent 只是它的副作用。

### Q19: SkillsManager 5 目录扫描顺序，为什么 workspace_dir 在前 cwd 在后？

**A**: 让 cwd 的同名技能覆盖 workspace 的——即"工作区覆盖系统"。`seen[skill["name"]] = skill` 是 dict 赋值，**后扫的覆盖先扫的**。语义：系统装了 v1，你想试 v2 就在 `cwd/skills/my-skill/` 放，下次启动生效。和 npm 的 `node_modules` vs 全局包类似。详见 [[06 - Intelligence#Q4]]。

### Q20: 为什么 `_hash_vector` 用哈希而不是真 embedding？

**A**: 教学版两个考量：(1) 不引入外部依赖（真 embedding 要么调 API 要么加载本地模型）；(2) 重点展示"双通道搜索的**模式**"（为什么 keyword + vector + decay + MMR），换真 embedding 时**管道结构不变**。哈希投影是 simhash 家族的近亲，用 hash 位的奇偶模拟随机投影。生产代码换 embedding，s06 学的是"为什么要 5 阶段"。详见 [[06 - Intelligence#Q6]]。

### Q21: 学 s06 要重点看哪几个函数？

**A**: 必读 5 个抽象层（共 ~250 行核心代码）：

1. `build_system_prompt`（L674，73 行）—— **本节灵魂**
2. `_auto_recall`（L831，6 行）—— 简单但信号量大：每轮都搜
3. `BootstrapLoader.load_all`（L142，20 行）—— 慢变量加载契约
4. `MemoryStore.hybrid_search`（L556，17 行）—— 5 阶段管道入口
5. `agent_loop`（L839，92 行）—— 启动加载 + 每轮重拼的串联

## 心跳与调度（s07 Heartbeat & Cron）

### Q22: 心跳和 cron 都是"定时跑"，本质差异是什么？

**A**: 两种完全不同的抽象。**Cron = 任务**（"9 点了跑日报"，无脑执行必有输出）；**心跳 = 唤醒**（"我醒了需要说话吗"，agent 自己决定沉默）。代码体现：心跳有 `_parse_response` 检查 HEARTBEAT_OK + `_last_output` 去重，cron 没这些。心跳是"让 LLM 拥有沉默权"，cron 是"按契约执行"。详见 [[07 - Heartbeat & Cron#Q1]]。

### Q23: lane_lock 为什么 cron 不参与？

**A**: 有意的不对称。心跳是探询式（被挡掉没损失，下次再问），cron 是承诺式（被挡掉就是漏跑，违背契约）。给 cron 加锁会破坏承诺语义。代价是 cron 多任务并发可能撞 LLM rate limit——生产代码在 CronService 内部加信号量解决，**不是用 lane_lock**。这是 s07 最容易看错的设计。详见 [[07 - Heartbeat & Cron#Q2]]。

### Q24: 心跳什么时候会真正说话？为什么教学版看不到效果？

**A**: 需要三件套齐备：(1) HEARTBEAT.md 写了真实指令（"如果 X 则报告"）；(2) agent 能访问外部数据源（邮件 / 日历 / 监控）；(3) 通过 IM 通道推送（Slack / 飞书）。教学版三件全无，所以 agent 每次只能 HEARTBEAT_OK。心跳的真实场景：邮件助手、长任务报警、日程提醒、PR review 摘要——共同特征是**用户不在场 + 外部状态变化 + IM push**。PuinClaw 接 Slack 后写真实 HEARTBEAT.md 才真正发光。详见 [[07 - Heartbeat & Cron#Q3]]。

### Q25: 心跳的输出为什么不进 agent_loop 的 messages？

**A**: 后台任务和用户对话的上下文必须隔离。心跳读 HEARTBEAT.md + SOUL.md + MEMORY.md 拼 prompt，**不读 messages**；输出**不入 messages**，只入 `_output_queue`。否则会污染——用户对话里突然插一条"心跳报告"，模型上下文乱掉。代价是心跳看不到用户最近聊了什么，但这是有意的取舍：心跳的语义是"全局巡更"，不是"对话补充"。详见 [[07 - Heartbeat & Cron#Q4]]。

### Q26: CronService 的 `every` 类型为什么对齐到 anchor？

**A**: 可预测性。不对齐（`now + interval`）会漂移——重启进程后第一次触发时间会变；对齐到 anchor 则重启后立刻按原计划继续。假设 `every=3600s, anchor=00:00`，无论何时启动，触发点都是整点。这是分布式调度的基本功。详见 [[07 - Heartbeat & Cron#Q6]]。

### Q27: 学 s07 要重点看哪几个函数？

**A**: 必读 5 处（共 ~120 行核心代码）：

1. `HeartbeatRunner._execute`（L205-230，26 行）—— **本节灵魂**：非阻塞抢锁 + 去重 + 入队
2. `HeartbeatRunner.should_run`（L170-187，18 行）—— 4 项前置检查契约
3. `CronService._run_job`（L405-454，50 行）—— 调度 + 错误自动禁用
4. `CronService._compute_next`（L363-390，28 行）—— 3 种调度类型核心
5. `agent_loop` 的 **L499（建锁）+ L503-504（传锁给心跳）+ L544-548（drain）+ L602/L641（用户路径加锁）** 共 ~10 行

跳过：`handle_repl_command`（调试命令）、`print_*` 辅助、croniter 库用法细节。详见 [[07 - Heartbeat & Cron#Q10]]。

### Q28: s07 和 learn-claude-code s14 的 cron 范式有什么根本差异？

**A**: 两种相反的架构选择。**s14（对称互斥）**：cron 通过 queue 投递、走 agent_loop 同一条路径、共享 messages，一把 `agent_lock` 让用户和 cron 排队。语义是"用户委托的延迟指令"。**s07（不对称）**：cron 直接调 `run_agent_single_turn`、独立 messages、不抢锁，一把 `lane_lock` 只在用户和心跳之间。语义是"agent 主动的告知行为"。分歧根源是目标场景：s14 是 CLI coding agent（对话连续性优先），s07 是 IM bot 前身（系统侧通知与用户对话解耦优先）。两个都不是错——学过 s14 再读 s07 容易误以为 claw0 设计有缺陷，实际上是为了 IM 场景刻意做的偏离。详见 [[07 - Heartbeat & Cron#vs learn-claude-code s14：两种 cron 范式]]。
