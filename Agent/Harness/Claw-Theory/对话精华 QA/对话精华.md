## 通道与并发（s04 Channels）

### Q1: `InboundMessage` 是 channel→agent 还是 agent→channel？

**A**: **channel → agent**。"Inbound" 站在 agent 视角——进入 agent 大脑的消息。出口方向（agent 回复）没有 `OutboundMessage` 类型，就是普通 `str` 传给 `channel.send(to, text)`。归一化成本只花在入站（入站结构复杂、出站简单）。

详见 [[04 - Channels#Q1]]。

### Q2: `run_agent_turn` 里 `messages.append({"content": inbound.text})` 看起来没把 Telegram 输入加进去？

**A**: `inbound` **就是** Telegram 消息——主循环 `for m in tg_msgs: run_agent_turn(m, ...)` 把后台线程解析好的 `InboundMessage` 传进来。`inbound.text` 已经是用户文字。到了这层，Telegram/CLI/飞书完全等价，这就是归一化的意义。

详见 [[04 - Channels#Q2]]。

### Q3: `ChannelManager.accounts` 和 `channels` 有什么区别？

**A**: 职责分离：

- `channels: dict[str, Channel]` —— **能干什么**（活的 receive/send 实例）
- `accounts: list[ChannelAccount]` —— **配了什么**（token / app_id 等配置记录）

`accounts` 在 s04 唯一用途是 `/accounts` REPL 命令打印 bot 列表。分开是为了未来支持"动态重载配置"。详见 [[04 - Channels#Q3]]。

### Q4: `run_agent_turn` 的 `else` 分支处理什么情况？

**A**: 兜底 `stop_reason` 不是 `end_turn` 也不是 `tool_use` 的情况——`max_tokens`（截断）、`stop_sequence`（停止词）、`pause_turn`（暂停）。模型可能已吐出部分文本但没说完，把这些片段发给用户然后 break（没法继续循环）。详见 [[04 - Channels#Q4]]。

### Q5: `select.select([sys.stdin], [], [], 0.5)` 是干什么的？

**A**: **非阻塞读 stdin** 的 POSIX 标准写法——把 stdin 当文件描述符 poll，最多等 0.5 秒，超时返回空列表。让主循环在"等 stdin 输入"和"排空 Telegram 后台队列"之间轮转。如果用阻塞 `input()`，主线程卡住，Telegram 队列堆着没人处理。详见 [[04 - Channels#Q6]]。

### Q6: s04 为什么要现场构造 `InboundMessage(text=user_input, channel="cli", ...)`？

**A**: 开了 Telegram 时主循环走非阻塞分支（`select`），只拿到裸字符串 `user_input`。手动包装成 `channel="cli"` 的 InboundMessage，让它走和 Telegram 完全相同的入口。没开 Telegram 时走 `cli.receive()` 直接返回 InboundMessage，就不需要这步。详见 [[04 - Channels#Q8]]。

### Q7: `for m in tg_msgs: run_agent_turn(m, ...)` 的串行限制是什么？

**A**: `run_agent_turn` 同步阻塞——10 条 Telegram 消息要串行处理 10 次，最后一条要等前 9 条处理完。这是 s04 的限制，**s10 Concurrency** 用"命名 lane"解决：不同 `peer_id` 走不同 lane 并行，同一 `peer_id` 仍串行（保证对话顺序）。详见 [[04 - Channels#Q9]]。

### Q8: 学 claw0 s04 要重点看哪几个函数？

**A**: 只看 6 个抽象层函数（`InboundMessage` / `build_session_key` / `Channel` ABC / `ChannelManager` / `run_agent_turn` / `agent_loop`）。`TelegramChannel` / `FeishuChannel` 内部的所有 `_buf_*`、`_flush_*`、`_parse`、`_refresh_token` 都是平台细节，**用到时再查**。真正的价值在 s05+（Gateway/Heartbeat/Delivery 这些通道无关架构）。详见 [[04 - Channels#Q11]]。

### Q9: claw0 的讲解是不是比 learn-claude-code 简化？

**A**: 数据反过来——claw0 单节均 doc 180 行 + code 747 行，learn-claude-code 是 115 + 330。差异在**风格**：claw0 是"参考手册"（是什么/怎么做），learn-claude-code 是"教程"（为什么/演进）。读 claw0 官方 md 时"为什么"部分得自己补。详见 [[04 - Channels#Q13]]。
