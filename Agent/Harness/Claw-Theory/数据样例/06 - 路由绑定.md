# 06 - 路由绑定（BindingTable）

> 来源：[`claw0/sessions/zh/s05_gateway_routing.py`](../claw0/sessions/zh/s05_gateway_routing.py)
>
> [[05 - Gateway & Routing]] 的 `BindingTable` 在内存里维护 5 级路由规则。**教学版不落盘**——bindings 在代码里硬编码或 REPL 命令动态加。OpenClaw 生产版会写到 `workspace/bindings.json`。

## Binding 的 5 个匹配维度

[[05 - Gateway & Routing]] 的 5 级路由优先级（T1 最严，T5 最松）：

| 级别 | 字段 | 含义 | 示例 |
|---|---|---|---|
| T1 | `peer_id` | 特定用户 / 群 | `"telegram:98765432"` |
| T2 | `guild_id` | 特定服务器 / 工作区 | `"discord:1234567890"` |
| T3 | `account_id` | 特定 bot 账号 | `"telegram:@my_bot"` |
| T4 | `channel` | 通道类型 | `"telegram"` / `"feishu"` / `"cli"` |
| T5 | `default` | 默认（兜底） | `"*"` |

**匹配规则**：从 T1 到 T5，**第一个匹配的就胜出**。详见 [[05 - Gateway & Routing#resolve_route]]。

## 字段语义

| 字段 | 类型 | 必填 | 含义 |
|---|---|---|---|
| `peer_id` | string? | 可选 | T1：特定用户匹配 |
| `guild_id` | string? | 可选 | T2：特定群组匹配 |
| `account_id` | string? | 可选 | T3：特定 bot 账号匹配 |
| `channel` | string? | 可选 | T4：通道类型匹配 |
| `agent_id` | string | ✓ | 匹配后路由到哪个 agent |
| `priority` | int | ✓ | 优先级（数字越大越优先） |

**字段省略 = 该维度不限制**。比如：

```json
{"channel": "telegram", "agent_id": "luna-default"}
```

表示"任何 Telegram 用户都路由到 luna-default"（不管 peer_id / guild_id / account_id）。

## 5 种典型 binding 配置

### 1. 个人专属（T1 严格匹配）

```json
{
  "peer_id": "telegram:98765432",
  "channel": "telegram",
  "agent_id": "luna-personal",
  "priority": 100
}
```

**语义**：Telegram 用户 `98765432` 的消息归 `luna-personal`。其他人不归这个 agent。

### 2. 群专属（T2）

```json
{
  "guild_id": "discord:1234567890",
  "channel": "discord",
  "agent_id": "team-assistant",
  "priority": 80
}
```

**语义**：Discord 服务器 `1234567890` 的所有消息归 `team-assistant`。

### 3. 多 bot 路由（T3）

```json
[
  {"account_id": "@luna_bot",    "channel": "telegram", "agent_id": "luna", "priority": 50},
  {"account_id": "@nova_bot",    "channel": "telegram", "agent_id": "nova", "priority": 50}
]
```

**语义**：同一 Telegram 通道挂两个 bot，用户 @luna_bot 走 luna，@nova_bot 走 nova。

### 4. 通道默认（T4）

```json
{"channel": "feishu", "agent_id": "feishu-helper", "priority": 30}
```

**语义**：飞书的所有消息归 `feishu-helper`（除非有更具体的 T1-T3 binding）。

### 5. 兜底（T5）

```json
{"agent_id": "default", "priority": 0}
```

**语义**：所有未匹配的消息归 `default` agent。**永远要有一条 default**——否则未匹配的消息会被丢弃。

## resolve_route 的实际匹配过程

用户在 Telegram（id=98765432）发消息给 @luna_bot，bindings 按顺序匹配：

```python
# s05_gateway_routing.py 的 resolve_route（简化）
def resolve_route(msg: InboundMessage) -> str:
    candidates = []
    for binding in self.bindings:
        score = 0
        if binding.peer_id    == msg.peer_id:    score += 1000  # T1
        if binding.guild_id   == msg.guild_id:   score += 100   # T2
        if binding.account_id == msg.account_id: score += 10    # T3
        if binding.channel    == msg.channel:    score += 1     # T4
        if score > 0 or binding.is_default:
            candidates.append((score, binding.agent_id))

    # 分数最高的胜出
    return max(candidates, key=lambda x: x[0])[1]
```

**例子**：用户 `telegram:98765432` 发消息：
- binding `{peer_id: "telegram:98765432", agent_id: "luna-personal"}` → score=1001
- binding `{channel: "telegram", agent_id: "luna-default"}` → score=1
- binding `{agent_id: "default"}` → score=0（兜底）

胜出：`luna-personal`（score 1001）。

---

## 为什么用 priority 而不是固定顺序

教学版用 priority 数字，**生产代码可能用顺序**。priority 的优势：
- 同 priority 的 binding 可以共存
- 调整优先级只改数字，不用调整列表顺序
- 易于动态添加（REPL 命令 +N 设优先级）

## 改造成 PuinClaw 时

PuinClaw 目前是**单 agent 单 bot**——不需要路由层。如果要扩成多 agent：

```python
# PuinClaw 多 agent 配置（推演）

    ├── luna/             # Luna 的 workspace

    │   └── ...
    ├── nova/             # Nova 的 workspace
    │   ├── SOUL.md
    │   └── ...

```
