# 04 - 投递队列（QueuedDelivery）

> 来源：[`claw0/sessions/zh/s08_delivery.py`](../claw0/sessions/zh/s08_delivery.py) L123-167
>
> [[08 - Delivery]] 的 `DeliveryQueue.enqueue()` 创建 QueuedDelivery → 原子写入磁盘（`.tmp.{pid}.{id}.json` → `fsync` → `os.replace`）→ 文件名就是 `{id}.json`。每条消息一个文件，存在 `outbox/` 目录。

## 目录结构

```
workspace/
└── outbox/
    ├── d4e5f6a7b8c9.json          ← 待发 / 已发的消息
    ├── e5f6a7b8c9d0.json
    ├── f6a7b8c9d0e1.json
    └── failed/
        └── a1b2c3d4e5f6.json      ← retry_count >= MAX_RETRIES 的失败消息
```

---

## 单个 QueuedDelivery JSON（待发状态）

```json
{
  "id": "d4e5f6a7b8c9",
  "channel": "telegram",
  "to": "98765432",
  "text": "你好！今天天气不错。",
  "retry_count": 0,
  "last_error": null,
  "enqueued_at": 1718956210.123,
  "next_retry_at": 0.0
}
```

## 失败重试中的 QueuedDelivery（retry_count=2）

```json
{
  "id": "e5f6a7b8c9d0",
  "channel": "telegram",
  "to": "98765432",
  "text": "这是第二天尝试发的消息",
  "retry_count": 2,
  "last_error": "TelegramApiError: 429 Too Many Requests",
  "enqueued_at": 1718956210.456,
  "next_retry_at": 1718956270.456
}
```

`next_retry_at` = `enqueued_at + BACKOFF_MS[retry_count]`（带 ±20% jitter）。详见 [[08 - Delivery#退避 + jitter：防雷群效应]]。

## 归入 failed/ 的 QueuedDelivery（retry_count >= 3）

```json
{
  "id": "a1b2c3d4e5f6",
  "channel": "telegram",
  "to": "98765432",
  "text": "这条消息重试了 3 次都失败",
  "retry_count": 3,
  "last_error": "TelegramApiError: Chat not found",
  "enqueued_at": 1718956210.789,
  "next_retry_at": 0.0
}
```

`retry_count >= MAX_RETRIES (3)` → 被 `move_to_failed` 移到 `outbox/failed/`。详见 [[08 - Delivery#retry_count 不重置：跟 s07 cron 的反差]]。

## 字段语义

| 字段 | 类型 | 默认 | 含义 |
|---|---|---|---|
| `id` | string | 必填 | 消息 UUID（文件名） |
| `channel` | string | 必填 | 目标通道（"telegram" / "feishu" / "cli"） |
| `to` | string | 必填 | 接收者 ID（Telegram chat_id / 飞书 open_id） |
| `text` | string | 必填 | 消息内容（已切块，单条不超过平台限制） |
| `retry_count` | int | 0 | 已重试次数（**只增不减**） |
| `last_error` | string? | null | 最近一次失败的错误消息 |
| `enqueued_at` | float | time.time() | 入队时间戳 |
| `next_retry_at` | float | 0.0 | 下次重试时间戳（0 = 立刻可试） |

**字段分组**：
- 3 个**内容字段**：`id` / `channel` / `to` / `text`（消息本身）
- 5 个**状态字段**：`retry_count` / `last_error` / `enqueued_at` / `next_retry_at`（队列管理）

## 状态流转

```
入队（retry_count=0）
    ↓
DeliveryRunner 发送尝试
    ↓
┌────────────────┬──────────────────┐
│     成功        │      失败         │
└────────┬───────┴────────┬─────────┘
         │                │
         ▼                ▼
   删除文件         retry_count += 1
   (ack)             last_error = 错误消息
                     next_retry_at = now + backoff[jitter]
                         │
                         ▼
                  retry_count >= MAX_RETRIES?
                         │
                ┌────────┴────────┐
                │ 否                │ 是
                ▼                 ▼
            留在 outbox/      move_to_failed
            等下次重试         → outbox/failed/
```

## 原子写入三件套（写入时的临时文件）

写入过程中**临时文件名带 pid 和 id**：

```
outbox/.tmp.{pid}.{id}.json   ← 写入中
                  ↓ fsync()
                  ↓ os.replace()
outbox/{id}.json               ← 写完改名
```

**为什么这么麻烦**——防止崩溃时产生"半写文件"。详见 [[08 - Delivery#原子写入三件套：崩溃安全的工程保证]]。

## outbox 目录的实际样子

```
$ ls -la workspace/outbox/

drwxr-xr-x  5 mr.puin  staff   160 Jun 23 14:30 .
drwxr-xr-x  8 mr.puin  staff   256 Jun 23 14:30 ..
-rw-r--r--  1 mr.puin  staff   142 Jun 23 14:25 d4e5f6a7b8c9.json     ← 待发
-rw-r--r--  1 mr.puin  staff   198 Jun 23 14:28 e5f6a7b8c9d0.json     ← 重试中
drwxr-xr-x  2 mr.puin  staff    64 Jun 23 14:30 failed
```

**DeliveryRunner 的工作**：每秒 `os.listdir(outbox)` → sort by `next_retry_at` → 逐个尝试发送。详见 [[08 - Delivery#DeliveryRunner 在 agent_loop 的位置]]。

## 多生产者写同一目录

3 个生产者（agent_loop / HeartbeatRunner / CronService）都会往 `outbox/` 写：

```
agent_loop（用户回复）       ──┐
HeartbeatRunner（心跳报告）  ──┼──► workspace/outbox/*.json
CronService（cron 输出）     ──┘
                                ↓
                        DeliveryRunner（单消费者）
```

**文件名 = UUID**，所以三个生产者同时写**不会冲突**——这正是用"每条一个文件"而不是"一个 queue.json 数组"的核心原因。
